# 単純な業務ロジックの実装（トランザクションスクリプト・アクティブレコード）

## 概要

業務ロジックはソフトウェアでもっとも重要な部分。業務領域のカテゴリーが異なれば、戦略的な重要性と複雑さが異なる。

この章では業務ロジックが**かなり単純な場合**に適した実装方法として、**トランザクションスクリプト**と**アクティブレコード**を扱う。

---

## トランザクションスクリプト（Transaction Script）（5.1）

> プレゼンテーション層からのリクエストを処理する手順の中に業務ロジックを埋め込む。
> — マーチン・ファウラー

システムが外部に公開するインターフェースは、システムの利用者が実行できる業務処理（トランザクション）の集まりと見なせる。それぞれのトランザクションはデータを参照したり更新したりする。

### 実装方法（5.1.1）

データ操作の手続きを、単純に順次処理式のスクリプトとして記述する。

- データベースを操作するために薄い抽象化レイヤーをはさむこともできる
- 処理手続きが満たすべき唯一の要件は**トランザクション管理**
- 処理は完全に成功または完全に失敗のどちらかで終了する（中途半端な状態で終了してはいけない）

```csharp
DB.StartTransaction();
var job = DB.LoadNextJob();
var json = LoadFile(job.Source);
var xml = ConvertJsonToXml(json);
WriteFile(job.Destination, xml.ToString());
DB.MarkJobAsCompleted(job);
DB.Commit();
```

### トランザクション管理の落とし穴（5.1.2）

見た目は単純だが、**トランザクション管理のまちがった実装**が多くの問題の原因になる。

**落とし穴1: トランザクション管理の欠落**

複数の更新処理を一つのトランザクションとして管理しないと、処理の途中で障害が発生した時にデータの不整合が起きる。

```csharp
// NG: 二つの操作がバラバラのトランザクション
_db.Execute("UPDATE Users SET last_visit=@p1 WHERE user_id=@p2", visitedOn, userId);
_db.Execute("INSERT INTO VisitsLog(user_id, visit_date) VALUES(@p1, @p2)", userId, visitedOn);

// OK: 一つのトランザクションとして管理
_db.StartTransaction();
try {
    _db.Execute("UPDATE Users ...");
    _db.Execute("INSERT INTO VisitsLog ...");
    _db.Commit();
} catch { _db.Rollback(); throw; }
```

**落とし穴2: 分散トランザクション**

複数データベースにまたがる操作、またはDBへの書き込みとメッセージ通信基盤への送出を組み合わせた操作は、分散トランザクションになる。

- 複雑な仕組みでスケールが難しく、エラーも起きやすい
- **多くの場合、分散トランザクションは使わない**
- 解決策は第8章（CQRS）・第9章（送信箱 Outbox）で扱う

**落とし穴3: 暗黙的な分散トランザクション**

一見、単一DBの単純な操作に見えても、クライアントと通信するだけで分散トランザクションになることがある。

例: `_db.Execute("UPDATE Users SET visits=visits+1 WHERE user_id=@p1", userId)` — 一つのDB更新だが、メソッドを呼び出したクライアントへの「結果通知」も含まれるため分散トランザクション。

- クライアントが結果を受け取れなかった場合にリトライすると、訪問回数が二重カウントされる
- 解決策の一つ: 操作を**べき等（idempotent）**にする（同じ操作を繰り返しても常に同じ結果になる性質）

### トランザクションスクリプトをいつ使うか（5.1.3）

**向いている場面:**
- 業務ロジックが手続き的なデータ操作だけという単純な問題領域
- データのETL（抽出・変換・書き出し）操作
- **補完的な業務領域**（業務ロジックが単純なのが定義）
- 一般的な業務領域に導入した外部サービスとの連係アダプター
- 区切られた文脈をつなぐ時のモデル変換装置の一部（第9章）

**使ってはいけない場面:**
- **中核の業務領域**（複雑な業務ロジックに対応できない）

**特性:**
- よい点: 単純さ。抽象化は最小限。実行時のオーバーヘッドは小さく、業務ロジックは理解しやすい
- 欠点: 業務ロジックが複雑になると、あちこちのスクリプトで同じ業務ロジックを重複して記述することになる。業務ロジックの一貫性が保てなくなり、不具合の原因になる

> トランザクションスクリプトはアンチパターンとされることさえあるが、広く採用されている実装方法。後の章で検討する実装方法は、何らかの形でトランザクションスクリプトが基礎になっている。

## アクティブレコード（Active Record）（5.2）

> データベースのテーブルまたはビューの1行を表すオブジェクト。データベース操作をカプセル化し、データに関連する業務ロジックを持つ。
> — マーチン・ファウラー

トランザクションスクリプトと同じく単純な業務ロジックの実装方法。

**トランザクションスクリプトとの違い:** 複雑なデータ構造（1対多・多対多の構造）を操作できる。

### 実装方法（5.2.1）

データ構造が複雑な場合に、アクティブレコードと呼ばれる専用のオブジェクトを使う。

- このオブジェクトはCRUD操作のメソッドを実装する
- ORM（Object-Relational Mapping）などデータベース操作のフレームワークと密結合する
- アクティブレコードという名称は「データ構造を表すオブジェクト自身が能動的（アクティブ）にデータを操作すること」に由来

アクティブレコードを使う場合も、業務ロジックは**トランザクションスクリプトとして**記述する。ただしトランザクションスクリプトはデータベースを直接操作するのではなく、アクティブレコードオブジェクトを操作する。

```csharp
public class CreateUser {
    public void Execute(UserDetails userDetails) {
        _db.StartTransaction();
        try {
            var user = new User();  // アクティブレコードオブジェクトの生成
            user.Name = userDetails.Name;
            user.Email = userDetails.Email;
            user.Save();  // DBへの永続化
            _db.Commit();
        } catch { _db.Rollback(); throw; }
    }
}
```

アクティブレコードの典型的な使い方は、**データ構造と振る舞い（業務ロジック）の分離**。たいていの場合、アクティブレコードがgetter/setterメソッドを公開し、外部の手続きがアクティブレコードのフィールドを変更する。

### アクティブレコードをいつ使うか（5.2.2）

アクティブレコードも本質的にはトランザクションスクリプト。主な目的はデータベース操作の最適化。

使う場合:
- **補完的な業務領域**
- **一般的な業務領域**用の外部サービスとの連係
- 区切られた文脈どうしを連係する時のモデルの変換

アクティブレコードは「貧血ドメインモデル」と呼ばれるアンチパターンとして知られているが、設計手法として否定的に扱うのは避けたい。業務ロジックが単純な場合にアクティブレコードを使うことに問題はない。むしろ業務ロジックが単純な場合、手の込んだ設計で実装するほうが不必要な複雑さを持ち込む。

> ここで説明しているアクティブレコードは**設計パターン**。ORMフレームワークのActive Recordではない。設計パターンとしてのアクティブレコードはマーチン・ファウラーが「エンタープライズアプリケーションアーキテクチャパターン」で紹介し、この設計パターンの実装方法の一つとして後から生まれたのがフレームワークとしてのActive Recordである。

## 比較

| 観点 | トランザクションスクリプト | アクティブレコード |
|---|---|---|
| データ構造 | フラット | 複雑（1対多・多対多） |
| DB操作 | 直接 | ActiveRecordオブジェクト経由 |
| 業務ロジック記述 | スクリプト内 | スクリプト内（ARオブジェクトを操作） |
| 向いている領域 | 補完・連係・ETL | 補完・外部サービス連係・モデル変換 |
| 向いていない領域 | 中核 | 中核 |

## 現実的に考える（5.3）

業務データは重要で、コードを書く時にデータの一貫性を保証しなければならない。しかし、現実的な選択が必要な場合もある。

大規模なデータ処理が必要な場合、データの一貫性に対して柔軟なアプローチを選択することもある。例: 百万件のレコードに対してたった1件の正しくないレコードを検出した場合、業務全体を停止すべきかどうかは事業による。

どんな状況にも適用できる汎用ルールはない。可能な範囲で特殊なケースを切り捨てることも選択肢の一つ。**特殊なケースを切り捨てることによるリスクと事業への影響を評価して設計判断をする**。

## 判断基準

**Q. トランザクションスクリプトとアクティブレコード、どちらを選ぶか？**

```
「データ構造は複雑か（1対多・多対多などのリレーション）？」
  YES → アクティブレコードを検討
  NO（フラットなデータ）→ トランザクションスクリプトで十分

「業務ロジックが複雑か（状態遷移・業務ルール・不変条件が絡み合う）？」
  YES → この章の実装方法は向いていない → ドメインモデル（第6章）へ
  NO  → トランザクションスクリプトまたはアクティブレコードを使う
```

**Q. どの業務領域に使うか？**

```
「中核の業務領域か？」
  YES → トランザクションスクリプトもアクティブレコードも使わない
       → ドメインモデルを使う
  NO  → 補完的か一般的な業務領域 → どちらも使える
```

## アンチパターン

**アンチパターン1: 中核の業務領域にトランザクションスクリプトを使う**
> 業務ロジックが複雑になると、あちこちのスクリプトで同じ業務ロジックを重複して記述することになる。業務ロジックの一貫性が保てなくなり、変更がやっかいで危険な大きな泥団子になる。

**アンチパターン2: トランザクション管理を省略する**
> 複数の更新処理を一つのトランザクションで管理しないと、処理途中の障害でデータの不整合が発生する。トランザクションスクリプトの名前の由来であるトランザクション管理は必須。

**アンチパターン3: 分散トランザクションを使う**
> 複数DBやメッセージ基盤にまたがる操作を分散トランザクションで管理しようとすると、複雑になりスケールが難しく、エラーも起きやすい。CQRS（第8章）や送信箱パターン（第9章）を検討する。
