# 境界づけられたコンテキスト（Bounded Context）

## この概念が答える判断

- 「同じ言葉が複数の意味で使われていて混乱している。どう切り分けるべきか」
- 「このモデル（またはspec）の境界はどこまで広げてよいか」
- 「業務領域（サブドメイン）の区切りと、実装単位の区切りは一致させるべきか」

## 原則

境界づけられたコンテキストとは、**ある言葉が一貫した意味を持ち続けられる範囲**のことであり、同時に**その意味を表現するモデルが実装として独立している範囲**でもある。

一つの言葉が事業のあらゆる場面で同じ意味を持つ、という前提は成り立たない。同じ言葉が部署・工程・利害関係者によって異なる指し示し方をされるのは、事業がある程度の規模を持てば自然に起きる現象であり、欠陥ではない。境界づけられたコンテキストは、この「暗黙のうちに複数存在している意味」を明示的な境界として切り出す手法である。

重要なのは、境界の**内側**でだけ言葉の一貫性を保証すればよく、組織全体で言葉を統一する必要はない、という点である。統一しようとすると、それぞれの文脈が本来持つべきモデルの独自性が失われる。

## 判断の分岐点

**Q1. ある言葉が複数の意味で使われている場合、どうするか**

- その言葉の使われ方が、関係者によって本質的に異なる捉え方をされている
  → コンテキストを分割し、それぞれの内側でその言葉に固有の意味を持たせる
- 単に表現が曖昧なだけで、本質的な意味は一つである
  → コンテキストを割るのではなく、用語自体を明確化する（分割は不要）

**Q2. コンテキストの大きさはどこまで広げてよいか**

言葉の一貫性が保てなくなる直前が上限。それを超えて広げると、モデルとしての実用性が失われる。
逆に、開発チームを分割する必要がある・非機能要件が異なる・スケーラビリティ要件が独立しているといった事情がなければ、無理に細分化する必要はない（過剰な分割は、組み合わせて全体を成立させる手間を増やすだけになる）。

**Q3. サブドメイン（業務領域）の区切りと、コンテキストの区切りは一致させるべきか**

サブドメインの分類（中核/一般/補完）は「事業がどう差別化を図ろうとしているか」という**事業側の発見**であり、コンテキストの境界は「それをどう実装単位に落とすか」という**設計側の判断**である。この二つは目的が異なるため、1対1に対応させることが最善の場合もあれば、そうでない場合もある。

判断の軸になるのは「同じデータに対して操作を行う一連の業務操作（ユースケース）が、複数のコンテキストに分散していないか」という点である。分散していれば設計の見直しが必要であり、分散していなければ1対1対応にこだわる必要はない。「一つのサブドメインには一つのモデル」という思い込みは持たない方がよい——一つのサブドメインの中に複数の異なる課題があれば、課題ごとに別のコンテキスト（別のモデル）を用意した方がうまくいくことがある。

**Q4. コンテキストと開発チームの関係はどうあるべきか**

一つのコンテキストの開発・変更・保守は、常に一つのチームが担う。複数のチームが一つのコンテキストを共同で担当すると、意図せず異なるモデルがそのコンテキストに混入していく。逆に、一つのチームが複数のコンテキストを担当することは問題ない。チームとコンテキストの対応は一方向（コンテキスト→チームは1つに決まるが、チーム→コンテキストは複数でもよい）である。

## 実例（架空の物流プラットフォーム）

集荷・配送・請求を一括で扱う物流プラットフォームを考える。当初「配送管理」と「配送状況の通知」という2つのコンテキストへの分割を検討したが、「同じデータ（配送1件の記録）に対する一連の操作が、意味もなく複数の区切りに分散していないか」を問うたところ、両者は一つの一貫した関心事（配送というモノの状態を追跡する）に属すると判断され、「配送管理」という単一のコンテキストに統合された。

一方で「請求・精算」は、配送管理とは扱う言葉も関心事も明確に異なる（Q1でいう「本質的に異なる捉え方をされている」に相当）——同じ「配送」という言葉でも、配送管理コンテキストでは「今どこにあるか」を指すのに対し、請求コンテキストでは「いくら課金する対象か」を指す。このため意図的に別のコンテキストとして切り分ける。

## アンチパターン

**組織全体で言葉を統一しようとする**
DDDにおける「同じ言葉」は組織横断の共通言語を意味しない。コンテキストの外側で言葉の一貫性を強制する必要はなく、無理に統一するとそれぞれの業務が本来持つモデルの独自性を損なう。

**複数のチームが一つのコンテキストを担当する**
一つのソフトウェアに、意図せず異なるモデルが混入する原因になる。コンテキスト間の連携が必要な場合は、暗黙の依存ではなく明示的な連携方針（通信規約）を定める。

**コンテキストを必要以上に細分化する**
分割しすぎると、それらを組み合わせて全体の振る舞いを成立させること自体が新たな難題になる。コンテキストの大きさは「モデルとして実用に足るか」を基準に判断する。

**サブドメインとコンテキストを常に1対1に固定する**
柔軟性を失う。一つのサブドメインに複数の異なる課題があるなら、課題ごとに別のコンテキストを用意した方がよい場合がある。
