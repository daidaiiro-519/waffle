# イベント駆動型アーキテクチャ（Event-Driven Architecture）

## この概念が答える判断

- 「複数のコンテキスト（サービス）が連係する必要がある。同期呼び出しではなく非同期のイベントで繋ぐべきか」
- 「イベントを発行するとき、どこまでの情報を載せてよいか。載せすぎると何が起きるか」
- 「連係が知らぬ間に密結合になっている気がする。何が原因で、どう直すか」
- 「イベント駆動型アーキテクチャとイベントソーシングは同じものか」

## 原則

イベント駆動型アーキテクチャ（EDA）は、**コンポーネント間で非同期にイベントメッセージをやり取りする通信方式**である。境界づけられたコンテキストの公開インターフェースを設計する手段の一つであり、内部実装の方式ではない。

これはイベントソーシング（集約の状態変化を内部でイベントとして記録する実装方式）とは範囲も役割も異なる。EDAは「コンテキストを越えてどう繋ぐか」、イベントソーシングは「コンテキストの内側でどう状態を持つか」を扱う。片方を採用したからといってもう片方を採用する必要はなく、両者は独立に選べる。

EDA設計の本質は、**どれだけの情報をイベントに載せるか**という一点に集約される。イベントに載せる情報量の選択（イベント通知／状態転送／業務イベント）が、コンポーネント間が疎結合になるか密結合になるかを直接左右する。載せすぎれば購読側は発行側の実装詳細に依存するようになり、少なすぎれば購読側は毎回問い合わせを強いられる。

## 判断の分岐点

**Q1. メッセージは「イベント」と「コマンド」のどちらとして設計すべきか**

- すでに起きた事実を伝えたいだけで、受け手に処理の可否を委ねる必要がない
  → **イベント**として設計する（名前は過去形にする）。イベントは一度発行されたら取り消せない。取り消しが必要なら、逆方向の新しいコマンドを発行する（補償アクション）
- これから行うべき操作を依頼するものであり、受け手が内容不備や業務ルール違反で拒否する余地を残したい
  → **コマンド**として設計する

**Q2. イベントにどこまでの情報を載せるべきか（3つのカテゴリーのどれを使うか）**

- 購読側に「何かが起きた」ことだけ伝えられればよく、発行側の内部状態を極力隠したい（セキュリティ上の理由・購読側が常に最新状態を問い合わせ直してほしい）
  → **イベント通知**（最小限の情報＋詳細を取得するための参照だけを載せる）
- 購読側が発行側の状態をローカルに複製し、発行側が落ちていても処理を継続できるようにしたい（結果整合性で十分・耐障害性を優先）
  → **イベントによる状態転送**（変更後の完全なスナップショット、または更新されたフィールドを載せる）
- 購読側の実装都合ではなく、業務領域で実際に起きた出来事そのものを記録・説明したい（他の購読者がいなくても意味を持つ）
  → **業務イベント**（その出来事を説明する情報を含めるが、発行側の内部状態をそのまま公開はしない）

**Q3. 密結合が疑われるとき、どの種類の結合か**

- 「Aの処理がBより先に終わっていないと成立しない」設計になっている
  → **時間的な結合**。原因は業務イベントによる状態転送に依存し、受信側が発行側の処理完了を前提にしていること
- 複数のコンポーネントが同じ業務ロジックを重複して実装しており、一方を直すと他方も直す必要がある
  → **機能的な結合**。同じイベントを購読し、同じ機能を車輪の再発明している
- 発行側のデータ構造を変えるたびに、購読側の処理が壊れる
  → **実装の結合**。イベントがそのまま内部モデルを晒しており、公開インターフェースとして独立していない

**Q4. 密結合をどう解消するか**

- 時間的な結合 → 「遅延（一定時間待つ）」で順序を保証しようとするのをやめ、イベント通知に切り替え、受信側が必要になったタイミングで発行側に問い合わせる設計に変える
- 機能的な結合・実装の結合 → 投影ロジック（受信側が本当に必要とする形へのデータ整形）を発行側にカプセル化し、その投影結果を公開インターフェース（コンシューマー駆動契約）として固定する。連係用のモデルを発行側の内部実装モデルから切り離す

**Q5. 一貫性の要求レベルに応じて、どちらを選ぶべきか**

- 結果整合性で十分（多少遅れて反映されても構わない）
  → イベントによる状態転送
- 強い一貫性が必要（受信側が発行側の最新の書き込みを必ず読む必要がある）
  → イベント通知を発行し、受信側が発行側に最新状態を問い合わせる

## 実例（架空の物流プラットフォーム）

集荷・配送・請求を扱う物流プラットフォームで、「配送管理」コンテキストが配送状況の変化を発行し、「通知」「請求」という別コンテキストがそれを購読する場面を考える。

まず、配送管理が発行するメッセージは「DeliveryStatusChanged」のようなイベントであり、名前は過去形にする。これは既に起きた事実であり、通知・請求のどちらの購読側もこのイベント自体を拒否することはできない（取り消したければ、配送管理の側が新しい訂正イベントを発行するしかない）。

次に、どこまでの情報を載せるかを検討する。

- 通知コンテキストは「配送状況が変わったこと」を利用者に知らせられればよく、詳細な荷物情報までは不要。**イベント通知**として最小限のペイロード（配送ID・新しいステータス・詳細への参照）だけを載せれば十分であり、配送管理内部の荷物データを外部に晒さずに済む。
- 請求コンテキストは「配送完了時点の重量・距離・時間帯」といった課金計算に必要なデータを継続的に自分側へ複製しておきたい（配送管理が一時的に落ちていても請求処理を止めたくない）。この場合は**イベントによる状態転送**として、課金計算に必要なフィールドをペイロードに含める。

ここで、もし配送管理が「配送完了に至るまでの内部処理ステップ」まで丸ごとイベントとして公開してしまうと、通知・請求の両方が配送管理の内部実装（社内システムのステータス遷移ロジック）に依存するようになる。これが**実装の結合**であり、配送管理側の内部フローを変更するたびに通知・請求側が壊れる。

また、「請求は配送完了イベントを受け取ってから、必ず通知側の処理が終わったあとに集計しなければならない」という順序前提を、待機時間の調整だけで担保しようとすると、これは**時間的な結合**であり、ネットワーク遅延や通知側の障害時に容易に破綻する。この場合は、請求側が「配送完了イベント（イベント通知）」を受け取った際に、必要になったタイミングで配送管理に最新状態を問い合わせる設計に直す。

## アンチパターン

**業務イベントをそのまま他コンテキストとの通信に使う**
発行側の内部状態を説明しきる情報を含む業務イベントを無制限に公開すると、購読側は実装の詳細に密結合する。公開する業務イベントを絞り込むか、イベント通知・状態転送に置き換える。

**遅延（待機時間）で時間的な結合を解決しようとする**
「一定時間待てば先行処理が終わっている」という前提は、ネットワーク遅延・高負荷・障害のもとで容易に破綻する。受信側が必要になったときに問い合わせる設計に変える。

**イベントカテゴリーを使い分けずすべてを同じ粒度で公開する**
イベント通知は連係の複雑さを減らすため、状態転送は非同期複製と耐障害性のため、業務イベントは業務領域そのものの記録のため、と目的が異なる。目的を考えずに一律の粒度を選ぶと、密結合と過剰な情報公開のどちらかに寄る。

**EDAとイベントソーシングを同じものとして扱う**
EDAはコンポーネント間の通信方式、イベントソーシングは境界づけられたコンテキスト内部の実装方式であり、独立に採否を決められる。「イベントを使っているから同じ設計」と考えると、内部実装の都合を外部インターフェースにそのまま漏らす原因になる。
