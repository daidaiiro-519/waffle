# 同じ言葉（ユビキタス言語 / Ubiquitous Language）

## この概念が答える判断

- 「この用語は業務エキスパートに通じる言葉になっているか、それとも技術者だけの隠語になっていないか」
- 「同じ綴りの言葉が、部署や工程によって微妙に違う意味で使われている。分けるべきか、そのままでよいか」
- 「会話・要件定義・テスト・コードで、言葉の使い方がずれてきた。何から手をつけるべきか」

## 原則

同じ言葉とは、業務エキスパートと開発者が同じ語彙・同じ言い回しで事業活動を語れる状態を指す。業務知識が要求分析担当者やプロダクト責任者といった「翻訳者」を経由して開発者に伝わるほど、伝聞のたびに意図が少しずつ欠落し、最終的に業務課題を解いていないソフトウェアができあがる。同じ言葉は、この翻訳の連鎖そのものを断ち切るための手法である。

同じ言葉が成立するための条件は次の3つに要約できる。

1. **業務用語のみで構成する** — シングルトンやファクトリーパターンのような実装都合の語彙を持ち込まない。業務エキスパートが理解できない語は同じ言葉ではない。
2. **一語一義（一貫性）** — 一つの用語には一つの意味しか持たせない。同じ綴りの言葉が文脈によって違う意味を指すなら、その言葉はすでに複数の概念を隠し持っている。
3. **同義語を作らない** — 同じ意味を指す語が複数あるなら一つに統一する。ただし、指しているものの振る舞いや責務が実際に異なるなら、あえて別語を立てるべきであり、これは「同義語の乱立」ではなく正しい書き分けである。

同じ言葉は一度整えれば終わりではなく、会話・要件・テスト・コードのすべての場所で同時に育て続ける対象である。

## 判断の分岐点

**Q1. この用語は同じ言葉として適切か**

- 業務エキスパートがその用語を使って業務を説明できない
  → 同じ言葉ではない。実装都合の語彙を業務語彙に置き換える
- その用語が文脈によって指す対象・意味が変わる
  → あいまいな用語。意味ごとに別の用語へ分解する（Q2へ）
- 同じ意味を表す別の用語がすでに存在する
  → 振る舞い・データ・責務に実質的な違いがあるかを確認する。違いがなければ一方に統一し、違いがあるならそれぞれを正式な別概念として定義する

**Q2. 一つの言葉が複数の意味を持っていると分かった場合、何を分けるべきか**

言葉そのものではなく、その言葉が指し示す**概念**を分ける。一つの語に複数の意味を握らせたまま実装すると、一つのクラス・一つのフィールドが相反する責務を同時に負うことになり、変更のたびにどちらの意味に対する変更かが曖昧になる。分けた結果、それぞれの意味が別の境界づけられたコンテキストに属することも多い。

**Q3. 会話とコードで言葉がずれてしまったら、どこから直すか**

一度に全体を統一しようとしない。まず自分たちが直接コントロールできる場所（ソースコードの命名・技術文書）から着手し、業務エキスパートとの会話の中で継続的にすり合わせる。同じ言葉の乱れは一度の作業で解消するものではなく、会話のたびに検出し修正し続けるものと捉える。

**Q4. 業務エキスパートの母語がコードの命名言語と異なる場合、どうするか**

無理に業務エキスパートの母語をそのままコードの識別子にする必要はない。実装上扱いやすい言語（多くの場合英語）でクラス名・フィールド名を定め、業務エキスパートが使う母語の名称はドキュメント上で対応づけて併記する。重要なのは識別子の文字種ではなく、両者が指している概念が一対一で対応していることである。

## 実例（架空のオンライン書店）

オンライン書店で「注文（Order）」という言葉を例に考える。

マーケティング担当者が「注文」と言うとき、それは主に「顧客が購入の意思表示をした事実」を指す——キャンペーンの効果測定やレコメンドの起点として関心があるのは、注文が確定した瞬間そのものである。

一方、物流担当者が「注文」と言うとき、関心があるのは「今どの倉庫にあり、いつ出荷され、いつ届くか」という状態の推移である。同じ「注文」という言葉を使っていても、前者にとって注文は確定した時点で完結する事実であり、後者にとって注文は出荷準備中・出荷済み・配達完了と状態を遷移し続けるモノである。

これはQ1でいう「文脈によって指す対象が変わる」あいまいな用語の典型であり、Q2に従えば、両者を無理に一つの「Order」概念に押し込めるのではなく、マーケティング視点の「注文（購入イベントの記録）」と物流視点の「出荷（配送状態を持つモノ）」を、それぞれ別の言葉として書き分けるべき状況である。どちらの言葉を採用するにせよ、少なくとも「マーケティング担当者が話す注文」と「物流担当者が話す注文」が同じ一語で指されている限り、開発者はどちらの意味でその語が使われているかを都度推測せざるを得ず、これ自体が同じ言葉が壊れているサインである。

## アンチパターン

**翻訳者を間に挟む**
業務エキスパートと開発者の間に要件定義担当者やプロダクト責任者を挟み、両者が直接対話しない状態。伝聞のたびに意図が変質し、同じ言葉が育つ機会そのものが失われる。

**コードだけを業務語彙に合わせる**
会話やドキュメントでは別の言い回しを使いながら、コードの識別子だけを業務用語にする。会話とコードの言葉がずれると、議論の内容とコードの振る舞いを突き合わせられなくなる。

**あいまいな用語をそのままモデル化する**
文脈によって意味が変わる用語に気づきながら、分解せずに一つのクラス・一つのフィールドとして実装してしまう。意味の衝突がそのままバグや設計の混乱として表面化する。
