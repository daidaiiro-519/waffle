# マイクロサービス

## この概念が答える判断

- 「このサービス（またはコンポーネント）の公開インターフェースは、どこまで小さくすべきか」
- 「マイクロサービスの境界は、区切られたコンテキスト・集約・サブドメインのどれに合わせるべきか」
- 「このサービス分割は、システム全体をかえって複雑にしていないか」

## 原則

マイクロサービスとは、**きわめて小さな公開インターフェースを持つサービス**のことである。ここでの「サービス」とは、規則として定められたインターフェースを通じて機能を提供する仕組みを指し、公開インターフェースはそのサービスの「玄関ドア」にあたる。

公開インターフェースを小さくすることには明確な利点がある。サービスが担う機能が明快になり、他のコンポーネントとのつながり方が理解しやすくなり、変更理由が限定されることで開発・運用・スケーリングを独立して行いやすくなる。マイクロサービスはデータベースを外部にさらけ出さず、連係に特化したコンパクトなインターフェースの内側にカプセル化する。

ただし、公開インターフェースを小さくすること自体は目的ではない。判断の中心にあるのは「**深さ**」という評価軸である。モジュール（サービス）は「機能（公開インターフェース）」と「ロジック（それを実現する内部の複雑さ）」の二つで定義できる。**深いモジュール**は、簡潔な公開インターフェースの奥に複雑なロジックをカプセル化しており、システム全体の複雑さを減少させる。**浅いモジュール**は、公開インターフェースがカプセル化する複雑さに見合わないほど大きく、逆にシステム全体の複雑さを増加させる。

複雑さには二種類ある。**局所的な複雑さ**（個々のサービス内部の複雑さ）と、**全体の複雑さ**（サービスどうしの相互作用・依存関係から生じるシステム全体の複雑さ）である。片方だけを最適化すると失敗する。全機能を一つのサービスにまとめれば局所的な複雑さは最大化する（大きな泥団子）。逆に公開インターフェースを極限まで細かくすれば、サービス間の連係のためにインターフェースが次々と拡張され、全体の複雑さが増加する（分散した大きな泥団子）。目指すべきは両方の複雑さを同時に最適化することである。

## 判断の分岐点（Q&A形式）

**Q1. マイクロサービスの境界と、区切られたコンテキストの境界はどう関係するか**

両者はしばしば混同されるが、非対称の関係にある。**すべてのマイクロサービスは区切られたコンテキストである。しかし、すべての区切られたコンテキストがマイクロサービスとは限らない。**

- 区切られたコンテキスト: そこに含まれるモデルの一貫性を守れる**もっとも広い**有効な境界を示す
- マイクロサービス: サービスとして成立する**もっとも小さい**有効な境界を定義する

境界を区切られたコンテキストの広さいっぱいまで広げると、単一のサービス（大きな一枚岩）になり得る。それ自体は問題ではなく、特定の状況では有効な選択である。合理的な設計の選択肢は、区切られたコンテキストの広さとマイクロサービスの狭さの間のどこかにある。

**Q2. 集約をマイクロサービスの単位にしてよいか**

集約はトランザクションの一貫性を守れる**もっとも狭い**有効な境界を表す。これは区切られたコンテキストと逆の性質であり、集約よりさらに細かい単位に物理的なサービスを分解することは、次善の策にすらならない。

集約を単独のマイクロサービスにする前に確認すべき点:
- 同じ業務領域内の他の集約と強くやりとりしているか
- 他の集約と値オブジェクトを共有しているか
- 集約の業務ロジックの変更が、業務領域の他のコンポーネントにどれだけ影響するか

これらの関係が強いほど、単独のサービスとしては「浅く」なりやすい。多くの場合、細かすぎる粒度のサービスはシステム全体の複雑さを増加させる。

**Q3. 結局、境界をどこに合わせれば安全か**

経験則として最もバランスが取れているのは、**サービスの境界をサブドメイン（業務領域）に合わせる**方法である。サブドメインは、事業活動の観点では「何をするか」を説明し、技術的観点では強く関連したユースケースの集合（同じモデル・密接なデータ・機能的に強い関係）である。この強い関係性が、結果としてモジュールの深さを保証する。ユースケースの集合をこれより小さい単位に割ると、公開インターフェースが複雑になりやすく、モジュールとして浅くなりやすい。

例外的に他の境界を選ぶ場合:
- 区切られたコンテキストの言葉の一貫性を広い範囲で保ちたい → 区切られたコンテキストの範囲で境界を設定する
- スケーリングやデプロイの独立性など非機能要件の要求が強い → 集約を境界として検討する（ただし多くの場合、サブドメイン単位の方が無難）

**Q4. サービス間の連係は、どう設計すれば公開インターフェースを小さく保てるか**

他のコンテキストの複雑さをそのまま公開インターフェースに持ち込むと、サービスは浅くなる。これを避ける典型的な設計が二つある。

- **共用サービス**: あるコンテキストのモデルと、他のコンポーネントと連係するためのモデルを分離する。連係のために限定された「公開された言葉（published language）」を外部に公開し、内部の実装モデルはいつでも入れ替えられる。
- **モデル変換装置**: 利用する側と提供する側の間に立ち、双方のモデルの変換を引き受ける。利用する側は変換の複雑さを意識せずに、使い勝手のよい連係専用モデルだけを扱える。

いずれも「連係のための複雑さ」を「業務ロジックの複雑さ」から分離し、サービスの公開インターフェースを簡潔に保つための手法である。

## 実例（架空の物流プラットフォーム）

配送・請求を扱う物流プラットフォームを、配送サービスと請求サービスの二つのマイクロサービスに分割する例を考える。

まず「配送サービス」を検討する。公開インターフェースとして「配送を登録する」「配送状況を更新する」「配送状況を問い合わせる」の3つのメソッドだけを外部に公開し、配送データベース自体は隠蔽する。内部では、集荷拠点の割り当て・配送員の稼働状況・再配達のロジックなど複雑な処理を行うが、外部からはそれが見えない。これは「深いサービス」の例である。

これを「1メソッド1サービス」の発想でさらに分解し、「配送を登録するサービス」「配送状況を更新するサービス」を別々の物理サービスにしたとする。この場合、両者は同じ配送データを扱うため、整合性を保つための同期処理や問い合わせ用のインターフェースが次々と必要になり、サービス間の呼び出しが増殖する。局所的な複雑さは下がったように見えても、全体の複雑さはむしろ増加する（分散した大きな泥団子）。

次に「請求サービス」を検討する。配送サービスと請求サービスは、Q1で述べた区切られたコンテキストの境界（「配送」という言葉の意味が両者で異なる）に沿っているため、無理なく別サービスとして独立させやすい。ただし請求サービスが配送の状態を都度問い合わせる設計にすると、配送サービス側の公開インターフェースが請求向けの特殊なメソッドで肥大化しやすい。これを避けるため、配送状況の変化を「配送完了イベント」として発行し、請求サービス側でそれを受け取って請求処理を進める設計にする。これはQ4で述べた「公開された言葉」の一種であり、配送サービスの内部実装がどう変わっても、請求サービス側の連係方法は変わらない。

このように、サービスの切り方そのものよりも「切った結果、公開インターフェースが小さく保てているか」「システム全体の複雑さが増えていないか」を確認することが判断の中心になる。

## アンチパターン

**「1サービスにつき1メソッド」を理想とする**
公開インターフェースを極限まで細かくすると、サービス間の連係のために公開メソッドが次々と追加され、分散した大きな泥団子になる。

**局所的な複雑さだけを最適化する**
個々のサービスを細かくするほど全体の複雑さが増大する。局所的な複雑さと全体の複雑さは両方をバランスよく最適化しなければならない。

**マイクロサービスと区切られたコンテキストを同一視する**
両者の関係は非対称である。すべてのマイクロサービスは区切られたコンテキストだが、逆は成り立たない。区切られたコンテキストがそのまま単一の大きなサービスになることも、状況によっては有効な選択である。

**集約を常にマイクロサービスの単位にする**
集約はトランザクション境界を表現する最小単位であり、それ以下への分解は望ましくない。同じ業務領域内の他の集約と強く関係している集約を無理に独立サービスにすると、浅いサービスになりやすい。

**論理的なモジュール分割を物理的なサービス分割と混同する**
一つのプロセス内で役割ごとにモジュールを分けることと、それらを別々のデプロイ単位に分割することは別の判断である。前者だけを行っている段階を「マイクロサービス化した」と称するのは不正確である。
