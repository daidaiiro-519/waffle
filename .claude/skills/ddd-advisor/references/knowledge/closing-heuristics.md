# 実装方法を選択する経験則

## この概念が答える判断

- 業務領域のカテゴリー（中核・一般・補完）が決まった後、業務ロジックの実装方法・土台となるアーキテクチャ・テスト戦略を具体的にどう選べばよいか
- 選んだ実装方法と、業務担当者が考えている業務領域のカテゴリーが食い違っているとき、どちらを疑うべきか
- 実装が「たいへん」だと感じたとき、それは単なる技術的な難易度なのか、それとも設計判断そのものを見直すべきサインなのか
- 区切られた文脈の境界を引き直すとき、どの粒度で区切るのが安全で、どの粒度が危険か

## 原則

実装方法の選択は、単発の判断ではなく**連鎖する判断**である。まず業務領域のカテゴリー（中核・一般・補完）を特定し、そのカテゴリーに応じて業務ロジックの実装方法（シンプルな実装か、本格的なドメインモデルか）を選び、その業務ロジックの実装方法に応じて土台となるアーキテクチャとテスト戦略が決まる。カテゴリーが変われば、この連鎖全体を見直す必要がある。

この連鎖のどの段階でも、判断の土台になるのは「同じ言葉（ユビキタス言語）」である。同じ言葉を早い段階から使って開発していれば、実装方法が現実に合わなくなった時点で気づきやすい。同じ言葉を欠いたまま設計を進めると、実装方法の破綻に気づくのが遅れ、後になるほど手直しの費用が大きくなる。

もう一つ押さえておくべき前提は、**業務領域のカテゴリーは固定ではない**という点である。ある機能が最初は補完的な業務領域として扱われていても、事業の発展とともに中核の業務領域へ性質を変えることがある。逆に、中核だと思われていた機能が、実際には競争優位に結びついていないただの補完領域だったと判明することもある。経験則は「一度カテゴリーを決めたら終わり」ではなく、「実装のたびに現在のカテゴリー認識が正しいかを問い直す」ための道具として使う。

## 判断の分岐点

**Q1. 業務領域のカテゴリーが決まった。業務ロジックの実装方法はどう選ぶか**

- 中核の業務領域である
  → 本格的なドメインモデルへの投資が正当化される。さらに、金額の扱い・分析・監査証跡が必要かどうかで、通常のドメインモデルにするか、変更履歴そのものを記録するイベント履歴式のドメインモデルにするかが分かれる
- 補完・一般の業務領域である
  → シンプルな実装（トランザクションスクリプトやアクティブレコードなど）で十分なことが多い。データ構造の複雑さや永続化モデルの数に応じて、その上に載せるアーキテクチャ（レイヤー数など）とテスト戦略（ピラミッド型かダイヤモンド型か等）を選ぶ

**Q2. 選んだ実装方法と、業務担当者の認識が食い違っている場合はどうするか**

これは「実装方法からカテゴリーを逆算して検証する」使い方である。実装方法を決めた後、それが本当にそのカテゴリーにふさわしいかを逆向きに確認する。

- 業務担当者は「中核の業務領域だ」と考えているのに、実装がごく短時間で終わってしまう
  → 開発側がその課題を過小評価しているか、その機能が実は競争優位に結びついていない可能性がある。業務担当者と話し合う
- 業務担当者は「補完の業務領域だ」と考えているのに、実装が本格的なドメインモデルやイベント履歴式の設計を必要とする
  → 業務担当者が要件を必要以上に複雑にしているか、逆にその機能が実は競争優位の源泉になっている可能性がある。カテゴリー認識自体を再検討する

**Q3. 実装が「たいへん」だと感じたら、どう扱うべきか**

実装の困難さを単なる技術的な壁として片づけず、事業の発展や設計判断の見直しにつながる重要なシグナルとして扱う。たいへんさに気づいたら業務エキスパートと話し合い、業務領域のカテゴリー認識と実装方法が今も整合しているかを確認する。業務ロジックを適切にモデル化できていれば、状況の変化に応じて実装方法を洗練させていくこと自体は、恐れるほど大きな手直しにはならない。

**Q4. 区切られた文脈の境界を、どの粒度で引き直すべきか**

境界の引き方には、安全さの異なる複数のパターンがある。

- 言葉による境界（同じ言葉が保てなくなる場所で区切る）→ 最も基本的で安全な境界の引き方
- 業務領域ごとの境界（一つの業務領域に一つの文脈を対応させる）→ 有効だが、業務領域とコンテキストの対応は必ずしも1対1でなくてよい（[[bounded-context]] Q3を参照）
- 集約ごとの境界（集約一つひとつを独立した文脈として扱う）→ 特定の状況でのみ有効。安易に適用すると、コンテキスト間の通信ばかりが増える「分散した大きな泥団子」を生みやすい
- 一つの集約を二つの文脈に分割する → 業務ロジックが引き裂かれるため、避けるべき境界の引き方

業務知識がまだ乏しい段階では、広めの文脈で始め、業務知識が増えるにつれて段階的に小さく分割していく方が、細分化しすぎる失敗を避けやすい。

## 実例（架空のオンライン書店）

在庫・注文・レコメンデーションを扱うオンライン書店を例にとる。

サービス立ち上げ当初、「注文管理」は業務ロジックがほとんどなく、在庫を引き当てて発送指示を出すだけだった。この時点では補完的な業務領域と判断し、シンプルなトランザクションスクリプトで実装した。同じ言葉（「注文」「引当」）を業務担当者と共有しながら開発していたため、後の変化に早く気づける土台ができていた。

事業が成長し、注文に対して複数の割引ルール・在庫の部分出荷・返品時の返金計算が絡むようになると、「注文管理」の実装が急激にたいへんになった。これは実装方法から逆に業務領域カテゴリーを検証すべきサイン（Q2・Q3）であり、実際には「注文管理」が競争優位を左右する中核の業務領域へと性質を変えていたことが判明した。そこでトランザクションスクリプトを、変更履歴を記録するイベント履歴式のドメインモデルへ置き換えた。

一方「レコメンデーション」は、当初「注文管理」と同じ文脈の中に実装されていたが、業務担当者との会話で「おすすめ」という言葉が、注文管理側では「よく一緒に注文される組み合わせ」を指し、レコメンデーション側では「行動履歴に基づく個人ごとの推薦」を指すという、本質的に異なる意味で使われていることが分かった。これは言葉による境界（Q4）が必要なサインであり、レコメンデーションを独立した文脈として切り出した。

## アンチパターン

**業務領域のカテゴリーを一度決めたきり見直さない**
業務領域のカテゴリーは事業の発展とともに変化しうる。補完だと思っていた機能が中核に育っていても実装方法を変えないままでいると、本来必要な複雑さに実装が追いつかず、無理な拡張を重ねることになる。

**業務担当者の認識と実装方法の食い違いを放置する**
実装方法から逆算したカテゴリーが業務担当者の認識とずれている場合、それはどちらかの理解が誤っているサインである。ずれを検出したまま実装を進めると、事業判断と技術判断が別々の前提で動き続けてしまう。

**実装の困難さを技術的な壁としてだけ処理する**
たいへんさを「頑張って乗り切るもの」として扱い、業務エキスパートと話し合う機会を作らないと、本来はカテゴリー変化のサインだったはずの兆候を見逃す。

**集約ごとにコンテキスト（あるいはサービス）を分割する**
「小さければ小さいほど良い」という思い込みから集約単位で境界を引くと、コンテキスト間の通信ばかりが増加し、疎結合を狙って分散した密結合（大きな泥団子が分散しただけの状態）になりやすい。

**一つの集約を二つの文脈に分割する**
業務ロジックが引き裂かれ、同じ業務ルールが複数の場所に重複して実装される危険が最も高い境界の引き方であり、避けるべきである。
