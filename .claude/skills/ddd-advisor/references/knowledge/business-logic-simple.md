# 単純な業務ロジックの実装（トランザクションスクリプト・アクティブレコード）

## この概念が答える判断

- 「この処理に、ドメインモデルのような手の込んだ設計を持ち込むべきか」
- 「業務ロジックがほとんど無い（単なるCRUD・単なるデータ変換）処理を、どう実装するのが適切か」
- 「複雑な設計パターンを使わないことは、手抜きなのか正しい選択なのか」

## 原則

業務ロジックの複雑さは、業務領域（サブドメイン）ごとに一様ではない。複雑な状態遷移や業務ルールが絡み合う領域もあれば、単純にデータを読み書きするだけの領域もある。実装方法は、扱う業務ロジックの複雑さに見合ったものを選ぶべきであり、あらゆる処理に同じ重厚さの設計を適用する必要はない。

業務ロジックが**手続き的なデータ操作の集まりにすぎない**場合、その実装は素直に「処理の手順をそのまま記述したスクリプト」でよい。これを**トランザクションスクリプト**と呼ぶ。スクリプトが唯一守るべき規律は、一連の更新処理が**完全に成功するか完全に失敗するか**のどちらかで終わること（トランザクション管理）であり、途中で中途半端な状態を残してはならない。

データ構造が単純なフラットな形ではなく、1対多・多対多のような複雑な関連を持つ場合は、そのデータ構造をカプセル化する専用のオブジェクト（**アクティブレコード**）を介して同じトランザクションスクリプトを書く。アクティブレコードは自分自身のCRUD操作を持つデータ構造であり、業務ロジックそのものを持つわけではない——業務ロジックは依然としてスクリプト側に置かれ、アクティブレコードはその操作対象を扱いやすくするための器にすぎない。

どちらの方法も、**業務ロジックが単純であること**を前提にした選択である。状態遷移・不変条件・業務ルールが絡み合う複雑な処理にこれらを適用すると、同じ業務ロジックがあちこちのスクリプトに重複して書かれ、一貫性が保てなくなる。その兆候が出たら、より表現力の高い実装方法（ドメインモデル）へ移行するべきタイミングである。

## 判断の分岐点

**Q1. データ構造は単純（フラット）か、複雑（1対多・多対多）か**

- フラットなデータで、関連するテーブル・オブジェクトをまたぐ必要がない
  → トランザクションスクリプトで十分。データ操作を直接手続きとして書く
- 1対多・多対多のような関連構造を持つ
  → アクティブレコードを検討する。データ構造をオブジェクトに包み、スクリプトはそのオブジェクトを操作する

**Q2. 業務ロジックは単純か、複雑か**

- 手続き的なデータの読み書き・変換だけで説明がつく
  → トランザクションスクリプトまたはアクティブレコードで実装してよい
- 状態遷移・業務ルール・不変条件が絡み合っている
  → この2つの実装方法は対応できない。ドメインモデルの採用を検討する

**Q3. この処理はどの業務領域に属するか**

- 事業の差別化要因である中核の業務領域
  → トランザクションスクリプトもアクティブレコードも使わない。ドメインモデルを使う
- 中核でない業務領域（一般・補完）
  → どちらの実装方法も選択肢になる

**Q4. トランザクション管理は正しく行われているか**

- 複数の更新をひとまとめの単位として扱えている（途中失敗時は全体がロールバックされる）
  → 問題ない
- 個々の更新がバラバラに実行されている、または複数のデータストア・外部システムをまたいで一つの操作として扱おうとしている
  → 前者はただちに修正が必要な欠陥。後者（分散トランザクション）は複雑さとエラー率が跳ね上がるため、多くの場合は避け、操作をべき等にする・イベント経由で連携するなど別の設計で解決する

**Q5. 「特殊なケースをどこまで丁寧に扱うか」に汎用ルールはあるか**

汎用ルールは無い。大量データを扱う処理では、ごく一部の不整合なレコードのために業務全体を止めるべきかどうかは事業判断による。特殊ケースを切り捨てることのリスクと、切り捨てないことのコスト（複雑さの増大）を比較して決める。

## 実例（架空のオンライン書店）

架空のオンライン書店における、在庫数の増減と注文ステータス更新を考える。

在庫更新は「注文が確定したら、対象書籍の在庫数を注文数量だけ減らす」という単純な手続きである。書籍と在庫数は1対1の単純な対応であり、業務ルールと呼べるものは実質的に存在しない。この場合、以下のようなトランザクションスクリプトで十分である。

```
BEGIN TRANSACTION
  order = LoadOrder(orderId)
  FOR EACH line IN order.lines:
    UPDATE Books SET stock = stock - line.quantity WHERE book_id = line.bookId
  UPDATE Orders SET status = 'CONFIRMED' WHERE order_id = orderId
COMMIT
```

一方、注文には「注文」「注文明細」「配送先住所」という関連する複数のデータ構造があり、これらをまとめて読み書きする場面が多い。この場合は各データ構造をアクティブレコードとして扱い、スクリプトはアクティブレコードのフィールドを操作してから保存する、という形に整理できる。

```
BEGIN TRANSACTION
  order = new Order()
  order.customerId = input.customerId
  order.lines = input.lines
  order.shippingAddress = input.shippingAddress
  order.status = 'PENDING'
  order.Save()
COMMIT
```

どちらの例でも、「在庫が引けるかどうかの与信判定」「注文者ごとの割引率の算出」のような、複数の条件が絡み合う業務ルールは登場していない。もしオンライン書店が将来「会員ランク・キャンペーン・在庫状況を組み合わせた動的な価格決定」のような複雑なルールを持ち始めたら、その部分だけドメインモデルに切り出すべきシグナルになる。

## アンチパターン

**中核の業務領域にトランザクションスクリプトやアクティブレコードを使う**
業務ロジックが複雑になるほど、同じ判断ロジックが複数のスクリプトに重複して書かれるようになる。重複した記述はやがて食い違い、一貫性の破綻や修正漏れの温床になる。

**トランザクション管理を省略する**
複数の更新処理を一つの単位として管理しないと、途中で障害が起きた際にデータが中途半端な状態のまま残る。トランザクション管理は「トランザクションスクリプト」という名前が示す通り、この実装方法の必須要件である。

**分散トランザクションに頼る**
複数のデータストアやメッセージ基盤をまたいで一つの操作として扱おうとすると、仕組みが複雑になり、スケールしにくくエラーも起きやすくなる。一つの更新に見えても、外部への通知を伴う操作は暗黙のうちに分散トランザクションになっていることがある点にも注意する。

**アクティブレコードだから設計として劣っている、と決めつける**
アクティブレコードは「貧血ドメインモデル」という文脈で否定的に語られることがあるが、業務ロジックが単純な場面でこれを使うこと自体は問題ではない。むしろ単純な業務ロジックに対して手の込んだ設計を持ち込むほうが、不必要な複雑さを生む。
