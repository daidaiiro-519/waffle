# コンテキストどうしの連係（Context Integration）

## この概念が答える判断

- 「複数の境界づけられたコンテキストが同じ業務を跨いで協調しなければならない。どういう関係として設計すべきか」
- 「上流・下流のどちらのモデルを優先すべきか、決定権をどう扱うか」
- 「連係するコストと、機能を重複させて連係しないコストのどちらが小さいか」

## 原則

境界づけられたコンテキストは独立して進化できるべきものだが、システム全体として機能する以上、コンテキストどうしは何らかの形で連係しなければならない。この連係の接合部分を「契約」と呼ぶ。契約が必要になるのは、異なるコンテキストは異なるモデル・異なる言葉を持つため、何も取り決めなければお互いの意図が正しく伝わらないからである。

連係の方法は無数にあるわけではなく、コンテキストを担当するチームどうしの**協力関係の性質**によっておおむね3つのグループに分類できる。

- 緊密に協力し合える関係（対等な協力）
- 一方が上流、一方が下流という力関係がある関係（利用者と供給者）
- そもそも協力しない、という選択（互いに独立）

どの連係方法を選ぶかは技術だけの問題ではなく、**チーム間の組織的な力学と協力の実態を反映した設計判断**である。連係方法を決めるときにコード上の実装しか見ないと、実際の協力関係と食い違ったパターンを選んでしまい、後で軋轢を生む。

## 判断の分岐点

**Q1. 二つのチームは緊密に協力できる関係にあるか**

- 両チームが同じ目標に強い意欲を持ち、頻繁に同期でき、片方が約束事を一方的に押しつけない
  → 「良きパートナー」を検討する。API変更は都度伝え合い、双方が協力して対応する関係
- さらに、どちらのコンテキストにも必須になる最小限のモデルが存在し、それを重複実装するコストが調整コストより高い
  → 「モデルの共有」を検討する。ただし共有範囲が広がるほど密結合になるため、共有するのは「契約と境界を越えて渡すデータ構造だけ」に絞るのが理想
- 物理的に離れている・組織の利害が衝突しているなど、緊密な協力が成立しない
  → Q2へ

**Q2. 利用者と供給者の関係において、決定権はどちらにあるか**

一方が上流（サービスを提供する側）、他方が下流（サービスを利用する側）になる関係では、決定権がどちらにあるかによって選択肢が変わる。

- 上流が決定権を持つ場合 → Q3へ（従属するか、モデル変換装置か）
- 下流が決定権を持つ場合 → 「共用サービス」を検討する。上流側が外部向けの公開インターフェースを内部モデルから切り離し、「公開された言葉」を提供することで、内部実装と外部契約を別々のタイミングで変更できるようにする

**Q3. 下流は上流のモデルをそのまま受け入れられるか**

- 上流の契約が業界標準的で安定しており、下流のニーズを十分に満たしている
  → 「従属する」を選ぶ。下流は上流のモデルをそのまま受け入れ、変換の手間をかけない
- 下流のコンテキストが中核の業務領域を含む、あるいは上流のモデルが使いにくい・頻繁に変更される
  → 「モデル変換装置（腐敗防止層）」を選ぶ。下流は自分たちのニーズに合わせてモデルを変換し、外部の概念を自分のモデルに持ち込ませない

**Q4. 連係のコストと重複実装のコストのどちらが小さいか**

- 組織が大きく部門間の合意形成に苦労する、あるいは対象が他社と変わらない一般的な業務領域で、既製の解決手段を各コンテキストが個別に組み込めばよい
  → 「互いに独立」を選ぶ。連係を諦め、必要な機能をコンテキストごとに重複させる
- ただし対象が中核の業務領域どうしの連係である場合、互いに独立を選ぶと、事業が最も力を入れて差別化すべき領域を重複実装することになり、事業戦略に反する
  → 中核領域の連係には互いに独立を使わない

## 実例（架空のレストラン予約プラットフォーム）

架空のレストラン予約プラットフォームを考える。「予約」コンテキスト（利用者の予約受付・空席管理を担当）と、「厨房オペレーション」コンテキスト（仕込み量の計画・調理の進行管理を担当）の二つがあるとする。

当初、両コンテキストのチームは頻繁に同期を取り合い、予約APIの変更があれば厨房側チームがすぐに合わせて対応する、という関係が成立していた。これは「良きパートナー」に相当する。しかし事業が拡大し、厨房オペレーションのチームが複数店舗展開先の別チームに分割されると、頻繁な同期が難しくなった。

そこで予約コンテキストが上流、厨房オペレーションコンテキストが下流という利用者・供給者の関係に整理し直した。予約コンテキストは「本日確定した予約一覧（人数・時間帯・コース種別）」という外部向けの契約を、内部の予約モデル（キャンセル履歴やクレジットカード情報などを含む複雑なモデル）から切り離して提供する「共用サービス」を用意し、厨房オペレーション側が使いやすい「公開された言葉」として公開した。

一方、厨房オペレーションのコンテキストが外部の食材発注サービス（他社が提供するSaaS）と連係する場面では、事情が異なる。厨房オペレーションは中核の業務領域であり、外部サービスの発注モデル（複雑なSKUコードや外部固有の単位系）にそのまま引きずられると、厨房側の「仕込み量」というモデルの表現がゆがんでしまう。そのため厨房オペレーション側に「モデル変換装置」を置き、外部の発注モデルを厨房側の言葉（食材・分量・仕込み単位）に変換してから取り込むことにした。

さらに、両コンテキストとも「ロギング基盤」のような一般的な業務については、共通サービスを作らずそれぞれが個別に既存のロギングフレームワークを組み込んでいる。これは「互いに独立」に相当し、中核でない一般領域だからこそ選べる選択である。

## アンチパターン

**中核の業務領域どうしの連係に「互いに独立」を使う**
中核の業務領域を重複して実装することは、その領域を最も効率的かつ最適化された方法で実装するという事業戦略に反する。

**中核の業務領域を含む下流が上流に「従属する」**
供給側のモデルに引きずられ、中核の課題を表現するはずのモデルがゆがんでしまう。この場合は「モデル変換装置」を使い、外部モデルを自分たちの言葉に変換してから取り込むべきである。

**モデルの共有範囲を広げすぎる**
共有されたモデルへの変更はすべての関係コンテキストに即座に影響する（密結合）。共有する範囲は「連係のための契約と境界を越えて渡すデータ構造だけ」に限定すべきであり、それ以上を共有すると、コンテキストが独立して進化できるという前提そのものが崩れる。

**連係方法をチーム間の協力の実態と切り離して選ぶ**
連係パターンは技術的な都合だけで決まるものではない。実際にはチームどうしの緊密な協力が成立していないのに「良きパートナー」を選ぶ、あるいは決定権の所在を確認せずに「共用サービス」と「モデル変換装置」を取り違えるといった選択は、いずれ組織的な軋轢として表面化する。
