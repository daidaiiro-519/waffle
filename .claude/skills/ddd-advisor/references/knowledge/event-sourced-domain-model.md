# イベント履歴式ドメインモデル（Event-Sourced Domain Model）

## この概念が答える判断

- 「集約の状態は最新値だけ保存すればよいのか、それとも状態が変化した経緯そのものを残すべきか」
- 「過去のある時点で集約がどんな状態だったかを、あとから正確に再現する必要があるか」
- 「監査ログ・分析・検索など、同じデータから複数の見方を導きたい場合、どう永続化を設計するか」

## 原則

イベント履歴式ドメインモデルとは、**集約の「今の状態」を直接保存するのではなく、状態を変化させた業務イベントの履歴を保存し、その履歴を順番に適用（投影）することで状態を再構築する**設計方式である。永続化されたイベント列そのものが「真実を語る唯一の情報源」（source of truth）になる。

従来のドメインモデルと部品（値オブジェクト・集約・業務イベント）は同じであり、対象領域も同様に複雑な業務ロジックを持つ中核の業務領域である。違いは永続化の対象が「状態」か「状態が変化した経緯」かという一点にある。

状態は、イベント列に対して変換ロジック（投影）を順に適用することで導出される。投影ロジックは一つに固定されない——同じイベント履歴から、最新状態を導く投影・検索用の索引を導く投影・分析用の集計を導く投影など、目的ごとに異なる投影ロジックを追加できる。これが「一つの情報源から複数の見方を導ける」という利点の根拠になっている。

## 判断の分岐点

**Q1. この業務領域にイベント履歴式ドメインモデルを使うべきか**

- 業務ロジックが単純で、対象が補完・一般の業務領域である
  → イベント履歴式は過剰。通常のドメインモデルより手前の方式（トランザクションスクリプト等）で十分
- 業務ロジックが複雑で中核の業務領域だが、状態の推移そのものに価値がない
  → 通常のドメインモデル（最新状態の保存）で十分
- 業務ロジックが複雑で中核の業務領域であり、かつ「過去の時点の状態の再現」「時系列の分析」「厳密な監査ログ」のいずれかが要求される
  → イベント履歴式ドメインモデルを検討する

**Q2. 監査ログや履歴はイベントストア以外の方法で代替できないか**

- ログファイルへの書き出しで代替する
  → データベースへの書き込みとファイルへの書き込みはトランザクション管理が別々になり、一方が失敗すると不整合が生じる
- 状態テーブルへのトリガーで履歴テーブルに複製する
  → 書き忘れは起きないが、記録されるのは「値がどう変わったか」だけで「なぜ変わったか」という業務の文脈は残らない。文脈がなければ、後から新しい投影ロジックを追加しても意味のある洞察を引き出せない
- 上記のいずれも要件を満たせない
  → 業務イベントそのものを最初から履歴として永続化する設計（イベントソーシング）を検討する

**Q3. イベント数の増大にどう備えるか**

- 一つの集約インスタンスに発生するイベント数が少ない（目安として数百以下）
  → 素朴に全イベントを読み込んで投影すればよく、追加の対策は不要
- イベント数が非常に多くなり、投影のたびに全履歴を読み込むと性能上の問題が出る
  → スナップショット（ある時点までの投影済み状態をキャッシュし、それ以降の差分イベントだけ適用する方式）を検討する。ただしスナップショットを検討する前に、まず集約の境界の設計そのものが適切かを見直すべきである（イベント数が肥大化するのは集約が大きすぎるサインであることが多い）

**Q4. 追記専用のイベントストアから、特定データを物理的に削除する必要が生じたら**

イベントの不変性（追記専用・削除不可）と、法規制等によるデータ削除要求は一見矛盾する。この場合、イベントそのものを消すのではなく、「イベントに含めた機微情報を暗号化し、復号キーを別のストアに保持しておき、削除要求があれば復号キーだけを破棄する」という間接的な方法で「実質的に読めなくする」設計が使われる。

**Q5. チームにイベントソーシングの実装経験がない場合、どう判断するか**

学習コストとモデル発展性（イベントのスキーマ変更は状態テーブルの変更より難しい）という欠点を、Q1で判断した必要性と天秤にかける。必要性が「あれば望ましい」程度なら、経験のないチームが最初から採用するのは推奨しない。

## 実例（架空の物流プラットフォーム）

配送1件のライフサイクルを、状態ではなくイベント履歴として持つ場合を考える。

```json
{ "shipment-id": "SHP-9001", "event": "集荷完了", "at": "2026-01-10T09:00:00Z" }
{ "shipment-id": "SHP-9001", "event": "輸送開始", "hub": "中継拠点A", "at": "2026-01-10T13:00:00Z" }
{ "shipment-id": "SHP-9001", "event": "拠点通過", "hub": "中継拠点B", "at": "2026-01-11T02:00:00Z" }
{ "shipment-id": "SHP-9001", "event": "配達完了", "at": "2026-01-11T15:30:00Z" }
```

このイベント列に対して、目的の異なる複数の投影ロジックを用意できる。

- **現在状態の投影**: 最後のイベントだけを見て「配達完了」という現在ステータスを導く
- **経路分析の投影**: 「拠点通過」イベントだけを集めて、この荷物がどの中継拠点を通ったかの経路を再構成する
- **リードタイム分析の投影**: 「集荷完了」と「配達完了」の時刻差を集計し、配送全体の所要時間の傾向を分析する

同じイベント履歴から複数の投影が導けること自体が、この方式の中心的な利点である。一方で、イベント数が増えるほど投影の計算コストは増し、イベントの構造（スキーマ）を後から変えることは状態テーブルの列を変えるより難しいという欠点も同時に持つ。

## アンチパターン

**補完・一般の業務領域にイベント履歴式ドメインモデルを使う**
「事業活動の視点で設計判断する」という原則から外れる。目の前の課題を解決するのに必要な理由がなく、より単純な方式で解決できる場合、学習コスト・技術的複雑さ・モデル発展性の低さといった欠点だけが表面化する。

**ログファイルを唯一の監査ログとして使う**
データベースへの書き込みとファイルへの書き込みのトランザクション管理が分離しているため、一貫性が保証されない。

**状態変更の「理由」を記録しない**
DBトリガーによる履歴複製は、項目の変化内容だけを記録し、変更の背景にある業務の文脈を失う。理由が記録されていなければ、後から新しい投影ロジックを追加しても有効な洞察を引き出せない。

**イベントのスキーマ変更を状態テーブルの列変更と同じ感覚で行う**
イベントは原則不変であり、一度永続化した構造を変えることは状態テーブルの変更よりはるかに難しい課題であることを軽視しない。
