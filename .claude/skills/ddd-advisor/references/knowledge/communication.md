# 通信（モデル変換・送信箱・サーガ・プロセスマネージャー）

## この概念が答える判断

- 「境界づけられたコンテキストどうしをどうつなぐか。内部モデルをそのまま外に見せていいか」
- 「業務イベントを他のコンテキストに確実に届けるには、どう発行すればよいか」
- 「複数のコンテキストにまたがる業務プロセスを、何を使って調整すべきか」
- 「その調整役はサーガでよいのか、プロセスマネージャーが必要なのか」

## 原則

境界づけられたコンテキストは内側で一貫したモデルを持つが、事業のプロセスはコンテキストの境界をまたいで流れる。この「境界をまたぐ通信」には、単体のコンテキスト設計とは別の原則が要る。

第一の原則は、**内部モデルをそのまま境界の外に露出しない**こと。コンテキストの内部で使う言葉・イベント・データ構造は、そのコンテキストの都合で自由に変わってよいものであり、それを外部にそのまま渡すと、外部のコンテキストが内部の実装詳細に依存してしまう。境界を越える際には、必ず変換の層を挟む。

第二の原則は、**集約の内部から直接メッセージ通信基盤へ発行しない**こと。集約はデータベースへのコミットで自らの状態変更を確定させる。イベントの発行とこのコミットが同一のトランザクションで保証されなければ、「コミットは失敗したのにイベントは届いた」「イベントは届く前にコミットが可視化された」といった不整合が生まれる。発行はコミットと不可分に結び付ける必要がある。

第三の原則は、**複数コンポーネントにまたがる業務プロセスの調整ロジックを、どのコンポーネントの内部にも埋め込まない**こと。調整ロジックはそれ自体が独立した責務であり、単純な直線的フローであれば軽量な仕組み（サーガ）で、複雑な分岐や状態管理を要するなら専用の調整役（プロセスマネージャー）として明示的に外出しする。

## 判断の分岐点

**Q1. コンテキストをまたぐ呼び出しで、送信元のモデルをそのまま送信先に渡してよいか**

- 送信元と送信先が同じ言葉・同じモデルを共有している
  → 変換不要でよいが、これは稀なケースであり、将来モデルが分岐した際に備えておく
- 送信元と送信先でモデルの意味・粒度が異なる（通常はこちら）
  → 境界に変換層を挟む。同期呼び出しなら呼び出しを横取りして変換する層（プロキシ）、非同期のメッセージングなら送信されるメッセージを横取りして変換する層（メッセージ変換器）を置く

**Q2. 変換に状態（複数データの保持・突き合わせ）が必要か**

- 1件のリクエスト・1件のメッセージを都度変換するだけでよい
  → 状態を持たない変換層で足りる
- 複数のリクエストやメッセージをまとめてから変換する必要がある、または複数の発生元からのデータを1つに組み立てる必要がある
  → 状態を持つ変換層（永続化を伴う集約・ストリーム処理）が必要になる

**Q3. 業務イベントをどう発行すれば、確実かつ安全か**

```
集約の内部から直接メッセージ通信基盤に発行してよいか
  → してはいけない（コミットとイベント発行が分離し、不整合の窓ができる）
更新された状態とイベントを同一トランザクションでデータベースにコミットし、
別の中継の仕組みが「まだ送信していないイベント」を検出して発行する
  → この形なら安全（送信箱パターン）
```

送信箱パターンは「最低1回は届く」ことを保証する仕組みであり、「ちょうど1回だけ届く」ことは保証しない。同じイベントが2回届く可能性を前提に、受け取る側は同じイベントを2回処理しても結果が変わらないように作る必要がある。

**Q4. 複数コンポーネントにまたがる業務プロセスを、サーガで実装してよいか、それとも専用の調整役が要るか**

```
そのプロセスにif-else的な分岐判断（複雑な業務ロジック）が必要か
  YES → プロセスマネージャー（状態を永続化し、次に何をすべきかを能動的に決定する）
  NO（特定のイベントを受けたら決まったコマンドを返すだけ） → サーガ（状態を持たない軽量な対応表でよい）
```

**Q5. コンテキストどうしの連携がうまくいかないとき、その調整の仕組み（サーガ等）を強化すべきか**

- 調整の仕組みそのものが力不足（イベントの取りこぼし、補償アクションの欠如など）
  → 調整の仕組み側を直す
- 実は「本来同じ集約に属すべき操作」が複数のコンポーネントに分散してしまっている
  → 調整の仕組みで無理に繋ごうとせず、集約の境界の設計を見直す。境界の内側は強い一貫性を前提にできるが、境界の外側は結果整合にしかならないため、本来1つの操作であるべきものを跨がせると、調整の仕組みがどれだけ精巧でも歪みが残る

## 実例（架空のレストラン予約プラットフォーム）

あるレストラン予約プラットフォームでは、「予約」「厨房」「会計」という3つのコンテキストが連携して1件の来店を処理する。

**モデル変換の例**: 「予約」コンテキストでは来店予定は「予約（Reservation）」という言葉で表現され、来店時刻・人数・卓の割当てを持つ。この予約が確定すると、「厨房」コンテキストに「仕込み指示（PrepInstruction）」として伝える必要があるが、厨房側は卓番号や予約者名には関心がなく、「何人前をいつまでに用意するか」だけを必要とする。両者の言葉が異なるため、予約確定イベントを横取りして仕込み指示に変換する層を境界に置く。予約の内部で使っている「予約ID」や「予約者の連絡先」をそのまま厨房コンテキストに渡さないことで、厨房側は予約コンテキストの内部モデルの変更に影響されずに済む。

**送信箱の例**: 「予約」コンテキストが予約確定を「厨房」「会計」の両方に伝える際、予約集約の内部から直接メッセージ通信基盤に発行するのではなく、「予約確定」という状態変更と「予約確定イベント」の記録を1つのトランザクションでデータベースにコミットする。別の中継の仕組みが未送信のイベントを検出し、厨房・会計それぞれに配信する。この仕組みにより、「データベース上は予約確定なのにイベントが飛んでいない」「イベントは飛んだがデータベースへのコミットは失敗していた」という状態を防ぐ。

**サーガの例**: 「予約が確定した」イベントを「厨房」が検知したら「仕込みを開始する」、「仕込みが完了した」イベントを「会計」が検知したら「請求書を発行する」——という単純な対応関係だけであれば、これは軽量なサーガとして実装できる。途中で仕込みが失敗した場合は、「予約を保留にする」という補償アクションを実行すればよい。

**プロセスマネージャーの例**: 一方、「団体予約かつ事前決済ありのコースメニュー」のように、予約内容によって「仕込みのタイミング」「決済の順序」「キャンセル時の返金可否」が分岐し、かつ複数のステップの進行状況を追跡し続ける必要があるなら、単純なイベント対応表では足りない。この場合は状態を永続化し、次に何をすべきかを能動的に判断する専用のプロセスマネージャーを用意する。

## アンチパターン

**集約の内部から業務イベントを直接メッセージ通信基盤に発行する**
データベースへのコミットと発行が別々の操作になるため、コミット前に発行してしまう、あるいはコミット失敗後でも発行済みという状態が起こり得る。状態とイベントを同一トランザクションでコミットし、別の中継の仕組みに発行を任せる。

**内部モデル・内部イベントをそのまま境界の外に公開する**
コンテキストの内部実装がそのまま外部の契約になってしまい、内部を変更するたびに外部が壊れる。境界を越える際は必ず変換層を挟み、外部向けの言葉に変換してから渡す。

**集約境界の設計不備を、調整の仕組み（サーガ等）で補おうとする**
本来同じ集約に閉じるべき一連の操作が複数のコンポーネントに分散している場合、どれだけ調整の仕組みを精巧にしても、境界の外側は結果整合にしかならないという制約から逃れられない。まず集約の境界を見直すべきであり、調整の仕組みで無理に繋いではいけない。

**単純な対応関係にプロセスマネージャーを使う／複雑な分岐にサーガを使う**
前者は過剰設計であり、後者は複雑な業務ロジックが「べき等な単純対応」の体裁の中に押し込まれて見えにくくなる。分岐の有無・状態管理の要否で機械的に選び分ける。
