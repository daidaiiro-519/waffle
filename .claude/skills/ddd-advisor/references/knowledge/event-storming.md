# イベントストーミング（Event Storming）

## この概念が答える判断

- 「業務プロセスの全体像を、参加者全員が同じ形で理解できていない。どう揃えるか」
- 「集約や境界づけられたコンテキストの区切りを、実装に着手する前にどう発見するか」
- 「業務エキスパートと開発者の間で、同じ言葉がまだ噛み合っていない状態をどう解消するか」

## 原則

イベントストーミングとは、**業務プロセスのモデルを短時間で作り上げるための協調型ワークショップ**である。成果物（できあがったモデルや境界の一覧）そのものより、**そのモデルを全員で一緒に作り上げる過程**に主な価値がある。

対象の業務領域に関わりうる関係者（開発者・業務エキスパート・プロダクトオーナー・テスター・UI/UXデザイナー・サポート担当など、知識を持つ人なら誰でも）が一堂に会し、大きな面（壁・オンラインボード）と色分けした付箋を使い、「実際に業務で何が起きているか」を時系列に沿って可視化していく。同じ面に向かって同じ言葉で議論すること自体が、ユビキタス言語を形成する場になる。

進め方は厳密なルールではなく指針であり、状況に応じて省略・簡略化してよい。ただし典型的な流れは、業務イベントの洗い出しから始まり、時系列整理、コマンド・アクター・集約の発見を経て、最後に境界づけられたコンテキストの候補を見出す、という順に情報を積み重ねていく点は共通している。

## 判断の分岐点（Q&A形式）

**Q1. どんな業務プロセスにイベントストーミングを使うべきか**

- 業務ロジックが複雑で、関係者の間でも認識が揃っていない、または誰も全体像を把握できていない
  → 実施する価値が高い。知識の分断を一枚のモデルに統合できる
- 手順が単純・自明で、関係者の理解にすでにズレがない（流れ作業に近い）
  → 実施の効果は薄い。時間と労力に見合わない

**Q2. 誰を、何人呼ぶべきか**

参加者は「その業務領域の知識を持つ人」全員が対象であり、エンジニアだけに閉じてはいけない——業務エキスパートが不在だと、技術的な視点だけに偏ったモデルになり、そもそもの目的（同じ言葉を作る）が果たせない。人数は、対面であれば全員が発言・参加できる規模、オンラインであればさらに絞った規模にとどめるのがよい。人数が増えすぎる場合は、複数グループに分けて実施し、後からモデルを統合する。

**Q3. どこまでのステップを一度にやるべきか**

組織にとって初めての試みであれば、いきなり全工程を通そうとせず、まず「業務イベントを洗い出し、時系列に並べ、注意点や転換点を見つける」ところまでで全体像を掴み、そのうえで個々の業務プロセスごとに残りの工程（コマンド・アクター・ポリシー・集約・境界の発見）を仕上げる、という二段階が無理がない。すでに手慣れているチームであれば、最初から一気通貫でよい。

**Q4. 境界づけられたコンテキストは、いつ・どうやって決めるべきか**

セッションの早い段階で境界を決め打ちしてはいけない。イベントストーミングの価値は探索にあり、境界は「機能的に密接に結びついた業務イベント・コマンドの集まり」が浮かび上がってきた**結果として**見出すものである。先に境界を固定すると、その枠に議論を押し込めてしまい、探索という本来の目的を損なう。

**Q5. モデリング要素（付箋の色分けなど）に厳密さを求めるべきか**

いいえ。色分けや記法は、参加者全員が「今何を書いているか」を瞬時に区別できるようにするための取り決めにすぎない。凡例をその場で共有し全員から見える場所に置いておけば十分であり、記法そのものを厳格に守ることが目的化してはならない。

## 実例（架空のレストラン予約プラットフォームのワークショップ）

架空のレストラン予約プラットフォームを運営するチームが、「予約」まわりの業務が複雑化し、エンジニアと店舗運営担当の間で用語や前提がずれてきたことを受け、半日のイベントストーミングを企画した。参加者はソフトウェアエンジニア2名、プロダクトオーナー1名、店舗側の業務エキスパート1名、サポート担当1名の計5名。

**発散的な洗い出し**では、参加者それぞれが思いつく限りの業務イベントを付箋に書き出した。「予約が仮登録された」「予約が確定された」「予約がキャンセルされた」「無断キャンセルが記録された」「席の割り当てが変更された」「来店確認が完了した」など、重複や粒度のばらつきを気にせず貼り出していく。

**時系列整理**の段階で、正常系（仮登録→確定→来店確認）を軸に並べたところ、店舗側の業務エキスパートから「確定後にも席の変更は頻繁に起きる」という指摘が出て、代替シナリオとして「席の割り当てが変更された」が正常系の枝分かれに追加された。ここで初めて、エンジニア側が想定していなかった業務知識（席変更の頻度と重要性）が可視化された。

**転換イベント**として、「予約が確定された」と「来店確認が完了した」の2つが浮かび上がった。前者は「見込み客の予約」から「確保された席」へ、後者は「確保された席」から「実績としての来店」へと、プロセスの性質が明確に切り替わる瞬間だったためである。

**コマンドとアクター**の洗い出しでは、「予約を確定する」というコマンドが、通常は店舗スタッフによって手動で実行される一方、一定条件下では自動で確定される場合があることが判明した。ここから、「仮登録から一定時間経過し、かつ空き状況に問題がなければ自動確定する」という**自動化ポリシー**が発見され、これは業務エキスパートも当初は意識していなかった暗黙のルールだった。

**集約の発見**では、「予約」に関する一連のコマンドとイベントが一つのまとまりとして扱えることが確認された一方、「無断キャンセルの記録」や「店舗の評価スコア算出」は、予約そのものの状態遷移とは異なる関心事として別枠に切り出された。

**境界づけられたコンテキストの候補**として、最終的に「予約管理」（仮登録〜来店確認までの一連の状態遷移）と「利用実績・評価」（無断キャンセル記録や評価スコアなど、予約が完了した後の分析的な関心事）の2つが自然な区切りとして浮かび上がった。この境界は、事前に誰かが決めていたものではなく、ワークショップを通じて参加者全員が「ここで関心事が変わる」と実感した結果として発見されたものである。

## アンチパターン

**最初から境界づけられたコンテキストを決めようとする**
イベントストーミングの目的は探索と知識共有にある。境界は工程の終盤で自然に浮かび上がるものであり、最初に境界を決め打ちすると、議論をその枠に押し込めてしまい、モデリング活動そのものの価値が失われる。

**エンジニアだけで実施する**
業務エキスパートなど、実務の知識を持つ関係者が参加しなければ、技術的な視点に偏ったモデルしかできあがらない。「同じ言葉を作る」という本来の目的も達成できない。

**業務イベントを現在形・未来形で書く**
業務イベントはすでに起きた出来事を表すため、必ず過去形で表現する。現在形や未来形で書くと、それがすでに起きた事実なのか、これから起きてほしい仕様なのかが曖昧になり、後工程の議論が混乱する。

**単純で自明な業務プロセスに対して実施する**
関係者の理解がすでに揃っている、複雑さのない業務領域にイベントストーミングを持ち込んでも、時間と労力に見合う発見は得られない。
