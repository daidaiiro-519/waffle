# データメッシュ（Data Mesh）

## この概念が答える判断

- 「複数の業務領域の分析データを、全社で一つの分析基盤にまとめるべきか」
- 「業務系（トランザクション処理）のモデルを、そのまま分析用途に転用してよいか」
- 「分析用データの品質・スキーマ・バージョン管理は誰が責任を持つべきか」
- 「区切られたコンテキストという考え方は、分析系データの設計にも適用できるか」

## 原則

データメッシュは、**分析系データに対するドメイン駆動設計の適用**である。境界づけられたコンテキストが業務系モデルの境界と保護を定義するのと同じ発想を、分析系モデルの所有権とスキーマの境界に適用する。

前提として、業務系モデル（トランザクション処理・OLTP）と分析系モデル（分析処理・OLAP）は**設計の目的が異なる**。業務系モデルはヒト・モノ・コトの相互作用を正確に記録・調整するために設計され、粒度は個々の記録単位で細かい。分析系モデルは事業活動のパフォーマンスに関する洞察を得るために設計され、集約されているほうが効率的に扱える。目的が違えば最適なモデルの形も違うため、**業務系モデルをそのまま分析目的に再利用することはできない**。

分析系モデルを構築する際によく使われる基本要素が、事実テーブル（Facts：何が起きたかを記録し、追記のみで変更・削除しない）と特性テーブル（Dimensions：事実の属性・性質を記述し、高度に正規化して柔軟な検索を支援する）である。この二種類のテーブルを事実テーブル中心に配置する構造が星形スキーマであり、特性テーブルをさらに細かく正規化したものが雪の結晶形スキーマである。

分析系データを扱う伝統的な基盤には二つの型がある。データウェアハウスは全社のデータを単一の分析系モデルに変換して集約する方式であり、データレイクは変換前の生データをいったんそのまま保管する方式である。どちらも「事業全体のあらゆる分析ニーズを単一モデル・単一基盤で満たそうとする」という同じ前提に立っており、この前提が規模の拡大とともに破綻する（データウェアハウスは業務系DBへの密結合でETLが壊れやすくなり、データレイクはスキーマ管理の欠如で「データの沼」と化す）。

データメッシュはこの前提を捨て、**単一の分析系モデルを目指さず、区切られたコンテキストごとに分析系モデルを分割して所有させる**。分析系モデルの所有権の境界は、業務系モデルの境界（＝境界づけられたコンテキストの境界）と自然に一致する。

## 判断の分岐点（Q&A形式）

**Q1. 業務系モデルを分析目的に再利用できるか**

- できない。業務系モデルはトランザクションの正確さと相互作用の調整のために最適化されており、分析系モデルは集約された洞察のために最適化されている。両者は別のモデルとして設計する
- 「同じデータだから同じモデルで扱えるはず」という直感は、業務系と分析系では最適化の方向が違うために成り立たない

**Q2. 分析基盤としてデータウェアハウス・データレイク・データメッシュのどれを選ぶか**

- 事業全体を単一の分析系モデルで扱いたい → データウェアハウス（ただし業務系DBへの密結合という根本課題が残る）
- 生データをいったん保管してから後で変換したい → データレイク（ただしスキーマレスによる品質劣化＝データの沼のリスクが残る）
- 区切られたコンテキストが複数存在し、かつモデルが頻繁に変化する（DDDに取り組んでいる） → データメッシュを検討する。密結合という根本原因そのものを解消しにいく選択だからである

**Q3. 分析系データを「プロダクト」として扱うとはどういうことか**

- 単にデータを提供するだけでなく、利用者が見つけやすい入口（データ出力ポート）・データの内容と形式を定義したスキーマ・信頼性を保証するSLA・破壊的変更を避けるバージョン管理を備えて初めて「プロダクト」と呼べる
- これらが欠けている状態は、分析系データを"プロダクトでなくただの副産物"として扱っている状態であり、データメッシュの原則に反する

**Q4. 区切られたコンテキストどうしの分析系モデルはどう連係させるか**

- 業務系モデルの連係方法（境界づけられたコンテキストの統合パターン）がそのまま分析系モデルにも適用できる
- 他のコンテキストが提供する分析系モデルをそのまま使えるなら連係方式を選び、モデルが合わなければ変換を挟むか、互いに独立したモデルを維持するかを判断する。この判断軸は業務系モデルの連係判断と同じである

**Q5. データメッシュの導入に必要な体制は何か**

- 各コンテキストのチームがデータプロダクトを作成・保守するだけでは足りない。相互運用性を支える共通管理基盤（アクセス方法・アクセス制御方針の統一）を整備する専任チームと、各コンテキストのデータオーナー・共通基盤チームの代表からなる統治体制（ルール作り）の両方が必要になる
- 体制を伴わずにモデルの分割だけを行うと、分割はしたが相互運用できない、という中途半端な状態に陥る

## 実例（架空のオンライン書店の販売分析データプロダクト）

架空のオンライン書店を考える。この書店には「注文管理」「在庫管理」「顧客サポート」という3つの境界づけられたコンテキストが存在する。

当初、経営陣は「全社の売上・在庫・問い合わせ状況を一つのダッシュボードで見たい」という要望から、3つのコンテキストの業務系DBから夜間バッチでデータを抽出し、単一の分析用データベースに変換して書き出すETLパイプラインを構築した（データウェアハウス方式）。運用を始めて分かったのは、注文管理チームが業務系テーブルのカラム名を変更するたびにETLスクリプトが壊れる、という問題だった。ETLが注文管理コンテキストの内部実装（テーブル構造）に直接依存していたためである。

この課題に対し、データメッシュの発想を導入する。注文管理コンテキストのチームは、自分たちの業務系モデル（注文というトランザクションの記録）とは別に、「注文の事実テーブル（注文ID・商品・数量・金額・発生日時）」と「顧客・書籍という特性テーブル」からなる星形スキーマの分析系モデルを設計し、これを安定したスキーマを持つ分析用エンドポイントとして他コンテキストに公開する。在庫管理・顧客サポートの各コンテキストも同様に、自分の分析系モデルを自分の責任で設計・公開する。

経営陣が見たい「全社ダッシュボード」は、3つのコンテキストが公開する分析系モデルを組み合わせて構築される。これにより、各コンテキストの業務系テーブルの内部変更がETLを壊す心配はなくなる（分析用エンドポイントというスキーマが変更に対する緩衝材になる）一方、3つの分析系モデルの整合性・アクセス方法を揃えるためのルール作り（どのコンテキストがどの粒度でデータを公開するか、SLAは何か）を担う横断チームが新たに必要になる。

## アンチパターン

**業務系DBをETLが直接参照する**
業務系システムのテーブル構造は内部の実装詳細であり、公開インターフェースではない。ETLがテーブルを直接参照すると、テーブル設計の些細な変更がETLを壊す。区切られたコンテキストが公開する分析用インターフェースを経由してデータを取得する。

**全社の分析データを一つのモデルに統合しようとする**
全社を一つの業務系モデルで表現しようとする失敗と同じ構図である。業務領域ごとに分析系モデルを分割し、コンテキストの境界と揃える。

**データレイクをスキーマレスのまま放置する**
取り込み時にスキーマ管理を行わないと、規模の拡大とともにデータの意味が誰にも分からない「データの沼」と化し、価値ある分析データを抽出するコストが跳ね上がる。

**分析系データをプロダクトとして管理しない**
明確な所有者・SLA・バージョン管理を欠いた分析系データは、品質と信頼性を維持できない。各コンテキストのチームが自分の分析データプロダクトに責任を持つ体制を整える。
