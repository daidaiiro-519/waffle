# ドメインモデル（値オブジェクト・エンティティ・集約・業務サービス）

## この概念が答える判断

- 「この概念は値オブジェクトか、エンティティか」
- 「このエンティティは集約の内部に持つべきか、IDで参照すべきか」
- 「複数の集約にまたがる計算ロジックはどこに書くべきか」
- 「そもそもドメインモデルを使うべきか、もっと単純な実装でよいか」

## 原則

複雑な業務ロジックを持つ中核の業務領域では、業務ルールをコード上に直接・明示的に表現する「ドメインモデル」という実装方法が適している。ドメインモデルは4つの部品からなる。

- **値オブジェクト**：IDを持たず、フィールドの値の組み合わせで同一性が決まる。状態は不変で、値を変える操作は新しいインスタンスを生成する。金額・住所・期間のように「それ自体は誰でもない」概念に使う。
- **エンティティ**：IDを持ち、状態が時間とともに変化する。ただしエンティティは単独では実装しない——必ず集約の一部として実装する。
- **集約**：データの一貫性境界。1つの集約は1つのトランザクションで変更される、という原則を持つ。外部の集約はIDで参照し、直接オブジェクトとして保持しない。
- **業務サービス**：値オブジェクトや集約単体では表現しにくい、あるいは複数の集約にまたがる業務ロジックを記述する。状態を持たない（ステートレス）。

この4部品の背後にある考え方は「不変条件をカプセル化することで複雑さを減らす」ことである。データが自由に変化できる度合い（自由度）が高いほどシステムは複雑になる。値オブジェクトと集約が、状態を変更できる経路を自分自身のメソッドだけに絞ることで、外側から見た自由度が下がり、振る舞いを予測しやすくなる。

## 判断の分岐点

**Q1. 値オブジェクトかエンティティか**

- 個々のインスタンスを個別に識別する必要がある → エンティティ（IDが必要）
- 同じ値であれば同じものとして扱える → 値オブジェクト
- 状態が時間とともに変化するか → 変化する場合はエンティティ（または集約）、変化しないなら値オブジェクト

**Q2. あるエンティティを集約の内部に持つか、IDで参照するか**

業務ロジックがそのエンティティの状態に強い一貫性（同一トランザクション内での整合性）を必要とするなら集約の内部に含める。結果整合性（多少の時間差があっても許容できる）で足りるなら、IDによる参照で外部に置く。集約はできるだけ小さく設計するのが経験則である。

**Q3. 業務サービスを使うべきか**

業務ロジックが単一の集約や値オブジェクトに閉じず、複数の集約を参照する計算であれば業務サービスを検討する。ただし「1集約=1トランザクション」の原則は業務サービスがあっても変わらない——業務サービスは複数集約を単一トランザクションでまとめて変更するための抜け道ではない。

**Q4. ドメインモデルをそもそも使うべきか**

業務ロジックが複雑で、その複雑さが事業の差別化に直結する中核の業務領域であれば、ドメインモデルを使う価値がある。業務ロジックが単純であれば、より軽量な実装方法（トランザクションスクリプト・アクティブレコード）で十分であり、無理にドメインモデルを適用する必要はない。

**Q5. 集約の設計に一貫性があるか**

集約の名前・フィールド名・メソッド名・発行する業務イベントの名前は、すべてその境界づけられたコンテキストの同じ言葉で命名されているべきである。名付けに一貫性が無い、あるいは自然な名前が付けられない場合、それは境界の引き方自体が間違っているサインであることが多い。

## 実例（架空の物流プラットフォーム）

配送1件を表す**Shipment集約**を考える。

- **値オブジェクト**：配送先住所（Address）。番地・郵便番号の組み合わせで意味が決まり、変更する時は新しい住所オブジェクトに置き換える。
- **エンティティ**：配送中に発生する「経由地点通過記録」。各記録は固有のIDを持ち、時刻・場所が更新されうるが、単独では存在せずShipment集約の内部にのみ存在する。
- **集約のルート**：Shipment自身。外部（アプリケーション層）はShipmentのメソッドを経由してのみ、内部の経由地点記録を追加・変更できる。
- **業務サービス**：複数の配送依頼をまとめて最適な配送ルートを計算する処理。単一のShipmentの状態には属さず、複数のShipmentと集荷担当者の空き状況を横断的に参照するため、業務サービスとして実装する。
- **業務イベント**：配送が完了した際に発行する`ShipmentDelivered`（過去形の命名）。この事象を他のコンテキスト（請求・通知）が参照する。

配送先住所の変更を許可すべきかどうかを検討した際、「配送中は住所変更に強い一貫性が必要か」という問い（Q2）に基づき、配送中の住所変更はShipment集約のメソッドを通じてのみ許可する、という設計判断がなされた。

## アンチパターン

**基本データ型への執着**
金額・住所・メールアドレスのような概念を単純な文字列や数値のまま扱うと、関連する業務ロジックがコードのあちこちに散らばる。値オブジェクトとして表現し、ロジックをその内部にカプセル化する。

**エンティティを単独で実装する**
エンティティは必ず集約の実装の一部として存在する。単独で実装すると、データの一貫性境界が定義できなくなる。

**複数の集約にまたがるトランザクション**
複数の集約を単一トランザクションでまとめて変更しようとする設計は、多くの場合、集約の境界の引き方自体が間違っているサインである。境界を見直すか、結果整合性で対処する。

**集約を大きくしすぎる**
集約が大きくなるほど、パフォーマンスや同時実行性の問題が起きやすくなる。強い一貫性が必要なデータだけを含め、できるだけ小さく設計する。

**業務イベントの名前を過去形にしない**
業務イベントは実際に起きた出来事を表す。現在形や命令形の名前は、業務イベントではなく業務コマンドを意味してしまう。
