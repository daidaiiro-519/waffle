# 設計の経験則（区切られた文脈の大きさ・実装方法・技術方式・テスト方針の総合判定）

## この概念が答える判断

- 「区切られた文脈（コンテキスト）はどれくらいの大きさで区切ればよいか」
- 「業務ロジックはどう実装すればよいか（トランザクションスクリプト／アクティブレコード／ドメインモデル／イベント履歴式ドメインモデル）」
- 「業務ロジックの実装方法が決まったとき、技術方式（レイヤードアーキテクチャ／ポートとアダプター／CQRS）とテスト方針（ピラミッド／ダイヤモンド／逆ピラミッド）はどう決まるか」
- 「サブドメインの分類（中核・一般・補完）と、実装方法・技術方式・テスト方針の選択に食い違いがあるとき、何を疑うべきか」

## 原則

これらは数学的に証明された定理ではなく、**経験則（ヒューリスティクス）**である。完全性は保証されないが、実務で判断の手がかりを絞り込むのに役立つ。大量の情報の中から「複雑さの根源」に関わる手がかりに焦点を当て、それ以外の雑音を遮断するための実践的な問題解決手法だと捉えるとよい。

経験則が扱う対象は次の2点に集約される。

- さまざまな業務活動に共通する本質的な特性（複雑か単純か、中核か周辺か）
- その特性に応じた設計判断（区切りの広さ・実装方法・技術方式・テスト方針）

これらは連鎖している。**区切られた文脈の大きさ→業務ロジックの実装方法→技術方式→テスト方針**の順に、前の判断が後の判断の前提になる。したがって、後段の判断に迷ったときは、前段の判断（特にサブドメイン分類）を疑い直すことが多くの場合の近道になる。

## 判断の分岐点

**Q1. 区切られた文脈はどれくらいの大きさにすべきか**

「最適な大きさ」という一般解は存在しない。大きさは結果であり、出発点ではない——**まずモデルを作り、その結果として境界の大きさが決まる**（逆に、先に大きさの方針（例:「小さいサービスにする」）を決めてからモデルを当てはめようとすると、モデルが歪む）。

判断の初期指針:
- 迷ったら**広い範囲を境界とするところから始める**。複数のコンテキストにまたがる変更は調整コストが高く、一度引いた境界の引き直しは重い（放置されがちになる）ため、最初から細かく割ることのリスクの方が大きい。
- 中核の業務領域を含むコンテキストを設計する場合は、その中核と**頻繁にやりとりする他の業務領域（中核・一般・補完のどれでもよい）を同じコンテキストに含めておく**。初期段階では予測しにくい変更にも対応しやすくなる。
- 業務知識が増え、変更の範囲や開発チームの体制が明確になった段階で、必要なら小さな文脈に切り分けていく。

**Q2. 業務ロジックの実装方法はどう選ぶか**

サブドメイン分類（→[[subdomain]]）を起点に、次の順で問う。

```
その業務領域は補完的・一般的な業務領域か？
  YES →
    扱うデータ構造は複雑か？
      YES → アクティブレコード
      NO  → トランザクションスクリプト

その業務領域は中核の業務領域か？
  YES →
    金銭処理・一貫した監査記録・業務視点での深い行動分析が必要か？
      YES → イベント履歴式ドメインモデル
      NO  → ドメインモデル
```

複雑さの見極めには、ユビキタス言語（→[[ubiquitous-language]]）の性質も手がかりになる。

- 会話が主にCRUD操作（登録・参照・更新・削除）の説明に終始する → 業務ロジックは単純
- 会話が入り組んだ業務プロセスや業務ルールの説明になる → 業務ロジックは複雑

**サブドメイン分類と実装方法が食い違っていないか**も点検する。中核と判断したのに実装方法がトランザクションスクリプトになっている、あるいは補完と判断したのに凝ったドメインモデルになっている——こうした不一致は実装のミスではなく、**サブドメイン分類そのものを見直す絶好の機会**である。

**Q3. 技術方式（アーキテクチャパターン）はどう決まるか**

実装方法が決まれば、技術方式（→[[architecture-patterns]]）は次のように対応する。

```
永続化モデルが複数必要か？
  YES → CQRSを追加

業務ロジックの実装方法は？
  イベント履歴式ドメインモデル → CQRS
  ドメインモデル              → ポートとアダプター
  アクティブレコード           → レイヤードアーキテクチャ（サービス層を含む4層）
  トランザクションスクリプト    → レイヤードアーキテクチャ（3層の最小構成）
```

対応の理由:
- イベント履歴式ドメインモデルはCQRSを要する。CQRSなしでは、集約の状態にもとづく参照系のクエリが極端に制限されるため。
- ドメインモデルはポートとアダプター（→[[architecture-patterns]]）を要する。レイヤードアーキテクチャでは、集約と値オブジェクトを永続化の都合から分離しにくいため。
- アクティブレコードはサービス層を足したレイヤードアーキテクチャに適する。サービス層がアクティブレコードの調整役を担うため。
- トランザクションスクリプトは最小構成のレイヤードアーキテクチャで足りる。

なお、CQRSは業務ロジックがイベント履歴式ドメインモデルでなくても、**複数の永続化モデルでデータを表現したい**という要求があれば単独で採用する価値がある。

**Q4. テスト方針はどう決まるか**

実装方法と技術方式が決まれば、テストにどこで比重を置くかも決まる。

| 方針 | 重点 | 向いている実装 |
|---|---|---|
| ピラミッド形 | 単体テストを厚く、統合・E2Eは薄く | ドメインモデル・イベント履歴式ドメインモデル |
| ダイヤモンド形 | 統合テストに重心 | アクティブレコード（業務ロジックがサービス層と業務ロジック層にまたがるため） |
| 逆ピラミッド形 | エンドツーエンド（処理フロー全体）を最重視 | トランザクションスクリプト（業務ロジックが単純でレイヤー数が少ないため） |

```
業務ロジックの実装方法は？
  トランザクションスクリプト   → 逆ピラミッド形
  アクティブレコード           → ダイヤモンド形
  ドメインモデル              → ピラミッド形
  イベント履歴式ドメインモデル → ピラミッド形
```

**Q5. この判定フローはどう使うべきか**

- あくまで経験則であり、絶対のルールではない。設計判断は常に文脈に依存する。
- 部分的に変更したり、独自の判定フローを組み立てたりしてよい。
- **できるだけ単純な手段を選ぶ**ことを基本にする。
- **複雑な手段（ドメインモデル・イベント履歴式ドメインモデル・CQRS）は、それが本当に必要なときの最後の手段として使う**。安易に高度な手段へ飛びつかない。

## 実例（架空のオンライン書店）

オンライン書店の「注文」に関わる業務領域を、経験則に沿って判定する。

まず「決済」は中核の業務領域と判断された（この書店は独自の後払い決済プランを差別化要因にしている）。中核なので、頻繁にやりとりする「注文」も同じコンテキストに含めることを検討し、モデルを育てながら実際に境界を引く（Q1）。

次に業務ロジックの実装方法を選ぶ（Q2）。決済は金銭処理そのものであり、一貫した監査記録も要求される→イベント履歴式ドメインモデルを選択する。一方、同じコンテキスト内でも「注文履歴の閲覧」は入力データの表示が中心で、ユビキタス言語も「一覧を返す」「詳細を返す」というCRUD的な説明に終始する→こちらは単純なトランザクションスクリプトのままでよいと判断し、決済の複雑な実装を注文履歴にまで引きずられて適用しない。

技術方式は実装方法から自動的に決まる(Q3)。決済（イベント履歴式ドメインモデル）はCQRSを採用し、注文履歴（トランザクションスクリプト）は最小構成のレイヤードアーキテクチャのままにする。テスト方針も同様に、決済はピラミッド形（単体テストで不変条件と金額計算を厚く検証）、注文履歴は逆ピラミッド形（画面表示までのE2Eを主軸に）とする(Q4)。

この例が示すのは、**同じコンテキストの中でも業務領域ごとに実装方法・技術方式・テスト方針が異なってよい**ということである。決済が中核だからといって、注文履歴にまで同じ重厚な設計を適用する必要はない。

## アンチパターン

**大きさから区切られた文脈を決める**
「マイクロサービスにするから小さく割る」「モノリスだから大きくまとめる」のように、実装形態の方針を先に決めてから境界を当てはめるのは順序が逆である。モデルが先、大きさは結果。

**すべての業務領域に同じ実装方法を使う**
チームがイベント履歴式ドメインモデルに習熟しているからといって、単純な業務領域にまでそれを適用すると、不要な複雑さと保守コストを持ち込むだけになる。業務領域ごとに経験則で使い分ける方が効率的である。

**補完的な業務領域に凝ったドメインモデルを使う**
業務ロジックが単純な補完領域に手の込んだ設計を持ち込むと、複雑さに見合わない保守コストだけが残る。

**サブドメイン分類と実装方法の不一致を放置する**
中核なのにトランザクションスクリプト、補完なのにイベント履歴式ドメインモデル、といった不一致に気づいても「まあいいか」と放置すると、後段の技術方式・テスト方針の選択もつられて不適切になる。不一致に気づいた時点でサブドメイン分類自体を疑い直す。
