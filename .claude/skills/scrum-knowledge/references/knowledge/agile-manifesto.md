## アジャイルソフトウェア開発宣言（4つの価値）

> 「私たちは、ソフトウェア開発の実践あるいは実践を手助けをする活動を通じて、よりよい開発方法を見つけだそうとしている。この活動を通じて、私たちは以下の価値に至った。」

| 左辺（重視する） | 右辺（価値があるが左辺を優先） |
|---|---|
| **個人と対話** | プロセスとツール |
| **動くソフトウェア** | 包括的なドキュメント |
| **顧客との協調** | 契約交渉 |
| **変化への対応** | 計画に従うこと |

> 「すなわち、左記のことがらに価値があることを認めながらも、私たちは右記のことがらにより価値をおく。」

## アジャイルソフトウェアの12の原則

### 原則1: 顧客満足を最優先とした早期・継続的な価値提供

顧客満足を最優先し、価値のあるソフトウェアを早く継続的に提供する。

### 原則2: 変化への歓迎

要求の変更はたとえ開発の後期であっても歓迎する。変化を味方にすることで顧客の競争力を高める。

### 原則3: 動くソフトウェアの短サイクル提供

動くソフトウェアを、2〜3週間から2〜3か月というできるだけ短い時間間隔でリリースする。

### 原則4: ビジネスと開発の日々の協働

ビジネス側の人と開発者は、プロジェクトを通して日々一緒に働かなければならない。

### 原則5: 意欲ある人々へのサポートと信頼

意欲に満ちた人々を集めてプロジェクトを構成する。環境と支援を与え、仕事が無事終わるまで彼らを信頼する。

### 原則6: フェイス・トゥ・フェイスによる情報伝達

情報を伝えるもっとも効率的で効果的な方法はフェイス・トゥ・フェイスで話をすることである。

### 原則7: 動くソフトウェアが進捗の最重要指標

動くソフトウェアこそが進捗の最も重要な尺度である。

### 原則8: 持続可能なペース

アジャイルプロセスは持続可能な開発を促進する。スポンサー・開発者・ユーザーは一定のペースを継続できるようにする。

### 原則9: 技術的卓越性と優れた設計への不断の注意

技術的卓越性と優れた設計に対する不断の注意が機敏さを高める。

### 原則10: シンプルさ

シンプルさ（ムダなく作れる量を最大限にすること）が本質である。

### 原則11: 自己組織化するチームから生まれる最良のアーキテクチャ・要求・設計

最良のアーキテクチャ・要求・設計は、自己組織化するチームから生み出される。

### 原則12: 定期的な振り返りと継続的改善

チームがもっと効率を高めることができるかを定期的に振り返り、それに基づいて自分たちのやり方を最適に調整する。

## Scrumとの対応関係

| アジャイル原則 | Scrumでの実践 |
|---|---|
| 原則1（早期・継続的な価値提供） | SprintごとのIncrement、Sprint Review |
| 原則2（変化への歓迎） | Product Backlogのリファインメント、Sprint中のスコープ交渉 |
| 原則3（短サイクル） | Sprint（最大1か月）の固定期間 |
| 原則4（日々の協働） | Daily Scrum、Sprint Review |
| 原則6（フェイス・トゥ・フェイス） | Daily Scrum（毎日同じ時間・場所） |
| 原則7（動くソフトウェアが指標） | Incrementの定義、Definition of Done |
| 原則12（定期的な振り返り） | Sprint Retrospective |
