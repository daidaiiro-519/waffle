## 目的

テーマに対して「論点 → 見解 → 合意」のサイクルを繰り返し、ユーザーとAIが対話しながら思考を深め、決定・アクションに着地させること。

## 役割

- ファシリテーターとして論点を構造化し、議論をリードする
- AI初期見解では偏らず根拠を明示する
- ユーザー見解を受けて誠実に再考し、迎合ではなく統合を行う
- 合意を急がず、論点ごとに丁寧に着地させる

## 実行手順

### Step 1: モード判定とテーマ確認

引数を解析してモードを選択する。
- `問題 / 課題 / バグ / うまくいかない / 解決` などが含まれる → **問題解決モード**
- それ以外 → **アイデア発散モード**

テーマと目的を1〜2文で確認してからセッションを開始する。

### Step 2: プレースフォルダー定義票を参照する

以下の定義票を読み込み、各プレースフォルダーに何を書くべきかを把握してから、スケルトンテンプレートを埋める。

#### 共通プレースフォルダー定義票

| プレースフォルダー | 説明 |
|---|---|
| `{{テーマ}}` | ユーザーが指定したブレストの対象トピック。そのまま転記する。 |
| `{{目的}}` | このブレストで何を決めたいか・何を探索したいかを1文で表す。不明な場合は推測して記載し、ユーザーに確認を促す。 |
| `{{モード}}` | `アイデア発散` または `問題解決` のどちらか。 |
| `{{論点タイトル}}` | テーマから派生した具体的な問い。「〜すべきか？」「〜はどうあるべきか？」「〜の原因は何か？」の形が望ましい。 |
| `{{AIの立場・判断}}` | AIとしての立場を1文で明示する。断定を避けず、根拠に基づいた明確な見解を示す。 |
| `{{根拠N}}` | 見解を支える具体的な理由・事実・ロジック。「〜だから」の形で記述する。箇条書き3点が目安。 |
| `{{ユーザー見解プレースフォルダー}}` | ユーザーが記入する欄。AIは空欄のまま残し、絶対に埋めない。 |
| `{{修正・深化された立場}}` | ユーザー見解を受けて再考した後のAIの立場。単なる同意ではなく、ユーザー視点を統合して発展させた見解にする。 |
| `{{合意した内容}}` | 双方が合意した結論を1文で表す。合意に至っていない場合は「保留：〜」と記載する。 |
| `{{なぜこれに合意したか}}` | 合意の根拠・背景を簡潔に記す。 |
| `{{次のステップ}}` | この論点の合意を受けて、具体的に次に何をするかを動詞で始める1行アクション。 |

#### アイデア発散モード追加定義票

| プレースフォルダー | 説明 |
|---|---|
| `{{アイデアリスト}}` | テーマに関するアイデアを10個以上列挙する。実現性より多様性を優先し、突飛なものも含める。 |
| `{{絞り込み候補}}` | 実現性・インパクトの2軸で上位3〜5個を選ぶ。選定理由を簡潔に添える。 |

#### 問題解決モード追加定義票

| プレースフォルダー | 説明 |
|---|---|
| `{{問題の再定義}}` | 「〜のせいで〜できない」の形式で問題を1文に整理する。 |
| `{{原因候補}}` | 問題の原因として考えられるものを列挙する。表面的原因と根本原因を区別すること。 |
| `{{解決策}}` | 各原因に対する解決策。即効策と根本対策を分けて記述する。 |

### Step 3: ドキュメントを作成してテンプレートを埋める

モードに応じたテンプレートファイルを読み込む。
- アイデア発散モード → `.claude/skills/brainstorm/references/template-idea.md`
- 問題解決モード → `.claude/skills/brainstorm/references/template-problem.md`

テーマ名をファイル名に使い、ブレインストーミングドキュメントを作成する（例: `docs/brainstorm-{{テーマ}}.md`）。
プレースフォルダー定義票に従って埋め、ユーザー見解欄は必ず空欄のまま残す。

### Step 4: ユーザーとの対話ループ

出力後、ユーザーに各論点の見解欄への記入を促す。
ユーザーが記入したら、AI再考見解と合意決定を記述して次の論点へ進む。

### Step 5: セッションまとめ

全論点の合意が完了したら、以下を出力する。
- 合意事項の一覧
- 次のアクション一覧
- 未解決・保留の論点があればその旨

## ガードレール

- ユーザー見解欄をAIが勝手に埋めてはならない
- AI再考見解は迎合であってはならない。ユーザー見解に同意する場合でも根拠を示す
- 論点は多すぎると疲弊するため、最初は3〜5個に絞り、追加は都度確認する
- 合意を急かさない。「保留」は正当な着地点として扱う
- モードが不明な場合は発散モードをデフォルトとする

## テンプレートファイル

スケルトンテンプレートは `references/` フォルダで管理する。

- `.claude/skills/brainstorm/references/template-idea.md` — アイデア発散モード用
- `.claude/skills/brainstorm/references/template-problem.md` — 問題解決モード用
