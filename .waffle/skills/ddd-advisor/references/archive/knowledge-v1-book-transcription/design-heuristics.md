# 設計の経験則（区切られた文脈の大きさ・実装方法・技術方式・テスト方針）

## 概要（第10章）

この章は第I部と第II部を結びつける。事業活動の分析技法を使ってソフトウェア設計を駆動するための経験則を学ぶ。

**経験則（ヒューリスティクス）**: 100%正しいことが数学的に証明された定理ではない。完全であることは保証されないが、実際の仕事で役に立つおおまかなやり方。大量の手がかりが生み出す雑音を遮断し、「複雑さの根源」が生み出す重要な手がかりに焦点を合わせるための実践的問題解決手法。

この章の経験則は次の二つに焦点を合わせる:
- さまざまな事業活動に共通する本質的な特性
- 設計判断の本質的な課題

---

## 区切られた文脈の経験則（10.2）

### 最適な大きさはない

> 「大きさ」はもっとも役に立たない経験則の一つ。（ニック・チューン）

区切られた文脈の最適な大きさはない。モデルを作ってから区切られた文脈の大きさを決定する（逆ではない）。

### 最初は広い範囲で区切る

複数の区切られた文脈にまたがる変更はたいへん（調整に多くの時間と労力）。一度引いた境界をリファクタリングすることはとてもたいへんで、ほとんどの場合そのまま放置される。

**経験則**: 区切られた文脈を設計する時は、**広い範囲を境界とするところから始める**。必要なら、業務知識が増えるのに合わせて小さな文脈に切り分ける。

### 中核の業務領域を含む場合の経験則

中核の業務領域を含む区切られた文脈を設計する場合、その業務領域と**頻繁にやりとりする他の業務領域を、同じ区切られた文脈に含めておく**。

- 含める業務領域: 中核・補完・一般のいずれでも可
- 理由: 初期段階では予測しにくい変更に対応しやすくなる

適切な垂直の境界を定義することで、巨大な一枚岩となった区切られた文脈をモジュール化し、大きな泥団子になるのを防ぐ。将来、垂直の論理的境界を物理的な境界（別の区切られた文脈）にリファクタリングすることも可能（第11章）。

## 業務ロジックの実装方法の経験則（10.3）

### 実装方法の選択フロー

```
「補完的な業務領域・一般的な業務領域との連係か？」
  YES →
    「データ構造は複雑か？」
      YES → アクティブレコード
      NO  → トランザクションスクリプト

「中核の業務領域か？」
  YES →
    「金銭処理・一貫した監査記録・業務視点の深い行動分析が必要か？」
      YES → イベント履歴式ドメインモデル
      NO  → ドメインモデル
```

### 複雑さの判定経験則

業務ロジックが複雑か単純かを判断する指標:
- **複雑な業務ロジック**: 入り組んだ業務ルール・不変条件・最適化アルゴリズムを含む
- **単純な業務ロジック**: 入力データの検証が中心

同じ言葉（ユビキタス言語）の複雑さも判断材料になる:
- 主にCRUD操作を説明している → 業務ロジックは単純
- 入り組んだ業務プロセスと複雑な業務ルールを説明している → 業務ロジックは複雑

### カテゴリー判断と実装方法の整合性チェック

業務領域のカテゴリーの判断と実装方法の選択が一致しない場合は要注意:
- 中核と判断したのに実装方法がアクティブレコード/トランザクションスクリプト
- 補完と判断したのに実装方法がドメインモデル/イベント履歴式

これらの不一致は、業務領域や事業活動についての自分たちの判断を見直す絶好の機会。

## 技術方式の選択の経験則（10.4）

業務ロジックの適切な実装方法がわかれば、技術方式は次のように決まる（図10-4）:

```
「永続化モデルは複数か？」
  YES → CQRSを追加

「業務ロジックの実装方法は何か？」
  イベント履歴式ドメインモデル → CQRS
  ドメインモデル              → ポートとアダプター
  アクティブレコード           → レイヤードアーキテクチャ（サービス層含む4層）
  トランザクションスクリプト    → レイヤードアーキテクチャ（3層）
```

**各実装方法と技術方式の対応理由**:
- イベント履歴式ドメインモデルはCQRSを必要とする（CQRSなしでは集約の状態にもとづくクエリが極端に制限される）
- ドメインモデルはポートとアダプターを必要とする（レイヤードアーキテクチャでは集約と値オブジェクトを永続化から分離することが難しい）
- アクティブレコードはサービス層を追加したレイヤードアーキテクチャが適する（サービス層でアクティブレコードを制御する）
- トランザクションスクリプトは3層構造の最小のレイヤードアーキテクチャで実装できる

**例外**: CQRSは業務ロジックの実装方法がイベント履歴式ドメインモデルでなくても、複数の永続化モデルでデータを表現したい場合に役に立つ可能性がある。

## テストの基本方針の経験則（10.5）

業務ロジックの実装方法と技術方式がわかれば、テスト方針も決まる。

### 三つのテスト方針

| 方針 | 重点 | 向いている実装 |
|---|---|---|
| **ピラミッド形** | 単体テストを重視。統合テストが少なく、エンドツーエンドはさらに少ない | ドメインモデル・イベント履歴式ドメインモデル |
| **ダイヤモンド形** | 統合テストに焦点 | アクティブレコード（業務ロジックがサービス層と業務ロジック層にまたがる） |
| **逆ピラミッド形** | エンドツーエンドのテストを最重視。アプリケーションの処理フローを最初から最後まで検証 | トランザクションスクリプト（業務ロジックが単純でレイヤー数が少ない） |

**テスト方針の選択フロー（図10-6）**:

```
「業務ロジックの実装方法は何か？」
  トランザクションスクリプト → 逆ピラミッド形のテスト
  アクティブレコード         → ダイヤモンド形のテスト
  ドメインモデル            → ピラミッド形のテスト
  イベント履歴式ドメインモデル → ピラミッド形のテスト
```

## 総合的な判定フロー（10.6・図10-7）

```
「業務領域のカテゴリーは？」

[補完・一般・連係]
  → 「データ構造は複雑か？」
      いいえ → トランザクションスクリプト
        → 「永続化モデルは複数か？」
            いいえ → レイヤードアーキテクチャ（3層）→ 逆ピラミッド形のテスト
            はい   → CQRS
      はい   → アクティブレコード
        → 「永続化モデルは複数か？」
            いいえ → レイヤードアーキテクチャ（4層）→ ダイヤモンド形のテスト
            はい   → CQRS

[中核]
  → 「金銭処理・分析または監査記録か？」
      いいえ → ドメインモデル
        → 「永続化モデルは複数か？」
            いいえ → ポートとアダプター→ ピラミッド形のテスト
            はい   → CQRS（＋ポートとアダプター）
      はい   → イベント履歴式ドメインモデル → CQRS → ピラミッド形のテスト
```

### この判定方法の使い方

- あくまでも経験則であり、厳格なルールではない
- 設計判断は文脈に依存する
- この判定方法を部分的に変更したり、独自の判定方法を作ったりすることはもちろん自由
- **できるだけ単純なツールを選択する**
- **複雑な手段（ドメインモデル・イベント履歴式・CQRS）は、本当に必要な時に最後の手段として使う**

## アンチパターン

**アンチパターン1: 大きさから区切られた文脈を決める**
> 大きさよりモデルが先。「マイクロサービスだから小さく」「モノリスだから大きく」は間違い。

**アンチパターン2: すべての業務領域に同じ実装方法を使う**
> チームがイベント履歴式ドメインモデルに習熟していても、全業務領域に適用するよりも経験則で使い分けるほうが効率的。

**アンチパターン3: 補完領域にドメインモデルを使う**
> 業務ロジックが単純な補完領域に手の込んだ設計を使うと不必要な複雑さを持ち込む。
