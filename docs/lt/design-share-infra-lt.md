# 補足資料 — 「API GatewayもLambdaも、使わないという選択。」

このドキュメントは `design-share-infra-lt.html`（LT資料 Part 1）の内容を、発表を離れて読んでも理解できるレベルまで掘り下げた補足資料です。スライドの番号・見出しに対応させて章立てしています。参照している実ファイルは `.waffle/skills/design-share/` 配下（`infra/cloudformation.yaml`・`infra/cloudfront-function/edge-gate.js`・`SKILL.md`・`scripts/ds.py`）です。

---

## 01. design-shareとは

design-share は、Claude Code というAIコーディングツール向けに、私（このリポジトリのオーナー）が自作した「Skill」です。

Claude Codeの「Skill」というのは、特定の目的のために「こういう手順で・こういう判断基準で作業しろ」という指示書（`SKILL.md`）と、それを実行するためのスクリプト群をひとまとめにしたパッケージです。ユーザーが自然言語で「UIモック作りたい」と言うと、Claude Codeがその指示書を読み込み、指示書に書かれた手順（対話でヒアリング → Design.md確定 → HTML生成 → デプロイ）を自律的に実行します。Skillの実体はプログラムではなく「AIが従うプレイブック」だと考えるのが近いです。

design-shareがやっていることは大きく2つです。

1. **Design.md** というデザインシステム定義（配色・タイポグラフィ・コンポーネントのトークン集）を、対話と試作の往復で確定する
2. 確定したDesign.mdから複数パターンのUIモックHTMLを生成し、それぞれを**トークン保護つきのURL**として自前のAWS環境に公開する

URLを開いた閲覧者はトークンを入力するだけでモックを見られ、ページ内蔵のコメント機能でその場に意見を残せます。アカウント登録は不要です。

---

## 02. なぜ作ったか（Artifactとの違い）

Claude Codeには「Artifact」という、チャット内で生成したHTMLやコードをその場にプレビューする機能が標準で備わっています。これは大変便利ですが、性質上「**今そのチャットセッションを開いている本人**」にしか見せられません。

具体的にはこういう制約があります。

- Artifactのプレビューは、そのチャットのセッション（ブラウザタブ／ウィンドウ）に紐づく
- POやデザイナーなど、Claude Codeのセッションを操作していない第三者に、そのプレビューをそのまま見せる手段がない（スクリーンショットを撮る、画面共有する、といった代替手段に頼ることになる）
- 仮に見せられたとしても、相手が直接「ここを直してほしい」とコメントを書き込む機能がない

design-shareが埋めているのはこの隙間です。design-shareが発行するのは**普通のURL**なので、Slackで送るなり、メールに貼るなり、好きな経路で第三者に渡せます。相手はブラウザだけでアクセスし、コメント機能でフィードバックを残せます。開発者側は溜まったコメントを管理コンソール（後述）で確認し、直して同じURLへ再デプロイする、というレビューのサイクルをそのまま回せます。

トークンは**パターン（＝生成したモックの案）ごとに1本ずつ**発行される設計です。これにより、あるURLのトークンが漏れても、他の案や他のプロジェクトのURLには影響しません。使い回しのある共通パスワードとは違い、被害範囲がその1パターンに閉じる、という設計判断です。

---

## 03. 課題

「UIモック、見といて」という一言の裏には、地味に面倒な作業が隠れています。

- Slackに画像を貼ると、フィードバックがスレッドの中に埋もれていき、後から「どの画像の話をしていたか」を追うのが辛い
- 複数案を出すと、「案A」「案B」がどの画像だったか分からなくなる
- バージョンが更新されるたびに、また画像を貼り直してスレッドが分岐する

これをちゃんと解決しようとすると、「URLを1つ発行して、そこにコメントも集める」という、実質的には小さなWebアプリケーションを1つ作ることになります。認証（誰でも見られては困る）、保存（コメントをどこかに残す）、コメントの投稿・一覧のAPI——これを都度スクラッチで作るのは割に合いません。design-shareは、これをSkillとして一度作ってしまうことで、以降は対話だけで済むようにした、という話です。

---

## 04. よくある構成

「認証つきで、ちょっとした状態（トークンやコメント）を持つWebアプリ」を素直にAWSで作ろうとすると、多くの人が思い浮かべるのはこの並びだと思います。

```
Browser → API Gateway → Lambda → DynamoDB
```

- **API Gateway**: HTTPリクエストを受け付け、認証・レート制限・ルーティングを行う入口
- **Lambda**: リクエストごとに起動する実行環境。認証ロジックやコメントのCRUDをここに書く
- **DynamoDB**: トークンやコメントを保存するNoSQLデータベース

これは決して間違った構成ではありません。ただし、この構成を採用すると同時に、以下のような管理対象が増えます。

- Lambdaの**コールドスタート**（久しぶりに呼ばれた関数の起動遅延）
- 各リソース間の**IAMロール設計**（Lambda実行ロール、DynamoDBへのアクセス権限など）
- **デプロイパイプライン**（Lambdaのコードをどう配布・更新するか）
- ランタイムのバージョン管理・依存パッケージの更新

design-shareが実際にやりたいことの粒度——「トークンで開ける」「コメントを溜める」「一覧を返す」——に対して、この装備が見合っているかを一度疑ってみた、というのが次の章の出発点です。

---

## 05. 実際の構成

design-shareが実際に採用しているのは次の3点です。

```
CloudFront（+ CloudFront Functions）
S3（private）
CloudFront KeyValueStore（DynamoDBの代わり）
```

Lambdaが要らなかった理由は単純で、「認証チェック」「無効化判定」「コメント投稿の形式検査」といった**軽いリクエスト単位の判定ロジック**を、Lambdaではなく **CloudFront Functions** という、CloudFrontのエッジ（利用者に地理的に近いロケーション）上で直接実行できる軽量な実行環境に載せたからです。そして、そのロジックが参照する「今有効なトークンは何か」という状態を、DynamoDBではなく **CloudFront KeyValueStore** というCloudFront専用のキーバリューストアに持たせました。

この2つの正体を、次の章から詳しく見ていきます（各サービスの内部構成のさらに詳しい話は、Part 2の補足資料で扱っています）。

## 06. 仕掛け① edge-gate.js（CloudFront Functions）

**CloudFront Functions** は、Lambda@Edgeよりもさらに軽量な実行環境です。JavaScript（ES5.1相当のサブセット）で書かれた小さな関数を、CloudFrontのエッジロケーションで、リクエストやレスポンスが通過するたびに実行できます。制約として、実行時間・メモリ・コードサイズ（10KB）が非常に小さく制限されており、代わりに**マイクロ秒オーダーの低レイテンシ**で動きます。ネットワークI/Oは基本的にできませんが、唯一の例外として**CloudFront KeyValueStore**への読み取りアクセスだけは許されています。

design-shareの `edge-gate.js` は、`viewer-request`（リクエストがオリジンへ届く前、CloudFrontに到着した直後のタイミング）で実行されるイベントに紐づけられており、次のことを1本のファイルで行っています。

1. **状態判定**: URLパス（`/p/{slug}/...`など）からパターンのslugを取り出し、KeyValueStoreに問い合わせて `token:{slug}` の現在値を取得する。値が `"DISABLED"` であれば無効化済み、キー自体が無ければ未発行のslugとして、いずれも403を返す
2. **トークン照合**: ブラウザが持っているCookie（`share_{slug}`）の値を、KeyValueStoreから取得した現在の有効トークンと突き合わせる。一致しなければ、トークン入力を促すHTMLページを返す
3. **トークン検証エンドポイント**: `/p/{slug}/verify` へのGETリクエストで、ヘッダー `x-share-token` に入っているトークンを検証し、成功すればCookieを発行する（204レスポンス）
4. **メソッド・書き込み検査**: 許可されているのは基本的に `GET`/`HEAD` と、コメント投稿用の `PUT /comments/{slug}/{id}.json` だけ。それ以外の変更系メソッド（`DELETE`/`POST`/`PATCH`など）は、経路を問わず一律で403を返す。これはS3のバケットポリシーだけに任せず、エッジの入口でも同じ制約を掛ける「多層防御」です
5. **コメント一覧の書き換え**: `/comments-list/{slug}` へのGETリクエストを、内部的にS3の `ListObjectsV2`（`prefix=comments/{slug}/`）というAPI呼び出し相当のクエリ文字列に書き換える。これにより「一覧APIをLambdaで実装する」ことなく、S3が本来持っている一覧機能をそのまま流用している

読み取り回数の設計にも配慮があります。通常、1回のアクセスで必要なKeyValueStoreの読み取りは1回（`token:{slug}`のみ）ですが、複数パターンを横断できる「ギャラリー」機能を使う経路では、所属するギャラリーの数（最大3つに制限）に応じて追加の読み取りが発生し、最悪ケースでも約6回に収まるよう設計されています。CloudFront Functionsには実行時間の予算があるため、この読み取り回数の見積もりは実務上重要です。予算を超えるとCloudFront Functions自体がエラーとなり、CloudFrontが5xxを返します——これは「わからないときは通す」ではなく「わからないときは拒否する」という **fail-closed**（安全側に倒す）設計です。

## 07. 仕掛け② CloudFront KeyValueStore

**CloudFront KeyValueStore（KVS）** は、CloudFront Functions専用に用意されたグローバルなキーバリューストアです。特徴は「CloudFront Functionsから直接、低レイテンシで読み取れる」ことで、DynamoDBのような汎用データベースをLambda経由で呼ぶよりずっと速く、かつCloudFront Functionsの実行制約の中に収まります。

design-shareが持っている状態は驚くほど少なく、次のキーだけです。

| キー | 用途 |
|---|---|
| `token:{slug}` | そのパターンの現在の有効トークン（値がそのまま失効判定にも使われる） |
| （値が）`"DISABLED"` | 無効化済みを示す特別な値 |
| `project:token` | 共有ギャラリー（全パターン横断閲覧）の共通トークン |
| `g:{gslug}` | 名前付きギャラリー（プロジェクト）ごとのスコープトークン |
| `pg:{slug}` | あるパターンがどのギャラリーに所属しているか（最大3件） |

テーブル設計もIAMロールの複雑な調整も要らず、「文字列のキーと値」だけで完結しています。

書き込み側（`scripts/ds.py`）の実装には、地味ながら重要な工夫があります。KVSへの書き込みは、まず `describe_key_value_store` でその時点のETag（版数のようなもの）を取得し、`put_key(..., IfMatch=そのETag)` という形で行われます。これは「自分が読んだときの版数のままであれば書き込む、途中で誰かが書き換えていたら失敗させる」という**楽観的並行性制御**です。ちょうどDynamoDBの条件付き書き込み（ConditionExpression）と同じ発想であり、「DynamoDBを使わない」と決めたはずの設計が、結局はDynamoDBが得意とする考え方を別の場所で再発明している、という面白い符合があります。

KVSはCloudFrontの全エッジロケーションに複製されるため、書き込んでから全世界のエッジに反映されるまでには数秒〜数十秒の伝播遅延があります。トークンのローテーションや無効化の直後は、この結果整合性の収束を待つ必要があり、design-shareでは「即時」とは案内せず「少し待って再読み込みしてほしい」と利用者に伝える設計になっています。

## 08. 仕掛け③ コメント機能

「コメント機能」と聞くとデータベースとAPIサーバーを連想しがちですが、design-shareの実装は驚くほど素朴です。

- **投稿**: ブラウザがブラウザ内蔵のJavaScriptから `PUT /comments/{slug}/{ランダムなID}.json` を叩く。中身はコメントのテキストなどを含むJSONそのもの
- **検査**: `edge-gate.js` が、このPUTリクエストのURLパスが期待する形（英数字のIDに `.json` が付いている）であること、`content-type` が `application/json` であること、サイズが16KB以内であることを検査する。この検査を通ったリクエストだけが実際にS3へ転送される
- **保存**: 実体は1コメント＝1つのS3オブジェクトとして、そのままバケット内に置かれる。データベースのテーブルではなく、ただのファイルの集まりです
- **一覧**: `GET /comments-list/{slug}` へのリクエストを、`edge-gate.js` がS3の `ListObjectsV2`（`prefix=comments/{slug}/`）へ書き換える。つまり「一覧を返すAPI」を自作するのではなく、S3が標準で持っているオブジェクト一覧機能をそのまま使っている

コメントは常にテキストとして描画され（`innerHTML`は使わない）、これに加えてエッジ側でも「JSON以外の形式のファイルをこのパスに置けない」ことを保証しているため、悪意あるHTMLファイルを配信ドメイン上に置かれてしまうリスク（stored XSS）を二重に防いでいます。

なお、S3バケットにはCORS設定を一切していません。これは意図的な選択で、コメントの読み書きは「配信ドメイン自身からの同一オリジンfetch」だけで完結するため、クロスオリジンでアクセスされる想定がそもそも無いからです。

## 09. Skillとの組み合わせ

ここまでのインフラの説明を踏まえて、実際にユーザーが体験するのはどういう流れかをまとめます。

1. チャットで「UIモック作りたい」と話しかける
2. design-share Skillが、内蔵のアーキタイプ（デザインの方向性の選択肢）を提示し、対話でヒアリングしながらDesign.mdを確定する。途中、配色やレイアウトの確認のために小さな試作を見せてくれる
3. 確定したDesign.mdから、コメント機能を内蔵した自己完結HTMLを複数パターン生成する
4. 生成したパターンをAWS環境へアップロードし、トークンを発行する——という一連の作業を、**Skill自身がコマンドを実行して行う**
5. 発行されたURLとトークンがチャット上にそのまま表示される。共有相手はそのままコメントで「この方向でOK」を返せる

ここで重要なのは、ユーザー（人間）は一度もコマンドを打っていないということです。AWSへの配置作業（アップロードやトークン発行）を実際に行っているのはdesign-share Skillそのもので、Claude Codeが指示書に従って裏側でコマンドを実行しています。ユーザーが最初から最後まで触れているのは、**自然言語のチャットだけ**です。コマンドもコンソールも、インフラの運用に関わる作業は、すべてSkill側に畳み込まれています。

---

## 10. まとめ

design-shareが証明しているのは、「サーバーレス＝API Gateway＋Lambda＋DynamoDB」という一つの型に固執する必要はない、ということです。

- **常時稼働コンピュートは0**——LambdaもEC2もコンテナも動いていない
- **課金対象のサービスは2種類**（CloudFrontとS3。KeyValueStoreはCloudFrontの一部として提供される）
- **アプリケーションロジックの実体はedge-gate.js 1本**に集約されている

「トークンで開ける」「コメントを溜める」「一覧を返す」——design-shareがやりたいことの粒度に、インフラの大きさを合わせただけです。大きな構成を最初に選ばない、という判断も設計の一部だという話でした。

各サービス（S3・CloudFront Distribution・Origin Access Control・各種ポリシー・CloudFront Functions・KeyValueStore・CloudFormation）の内部構成をさらに詳しく知りたい場合は、Part 2の補足資料（`design-share-infra-lt-part2.md`）を参照してください。
