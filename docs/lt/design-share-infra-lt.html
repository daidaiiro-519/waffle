01 design-shareとは

自作した、「UIモックを、URLで」渡すSkill

design-share は、私が Claude Code 向けに自作した Skill。「UIモック作りたい」とチャットで頼むと、 対話だけで見た目の合意から共有可能なURLまで面倒を見てくれる。

対話でヒアリング
→
Design.md確定
→
UIモックHTML生成
→
共有URL + コメント

途中で生まれるDesign.mdは配色・タイポグラフィ・コンポーネントを決めるデザインシステム定義。 確定後にそこから複数パターンのUIモックを生成し、パターンごとにトークン保護つきのURLを発行する。 関係者はURLを開いてトークンを入れるだけで見られて、コメントもその場に残せる——アカウント登録は要らない。

02 なぜ作ったか

Claude Code標準のArtifactは、他人と共有できない

Claude Codeにはその場でモックを見せる「Artifact」がある。便利だが、 今チャットしている自分のためのものだ。POにも、デザイナーにも、そのURLをそのまま渡せない。

URL＋トークンを渡す
→
相手がブラウザでコメント
→
コンソールで確認
→
直して再配布

design-shareが埋めているのはこの隙間。Artifactにはないコメント機能を持たせ、 URLとトークンさえ渡せば、Claude Codeを触ったことのない人でもレビューの輪に入れる。 しかもトークンはパターンごとに1本ずつ発行——使い回さないので、1件漏れても他は無関係のまま。

03 課題

「UIモック、見といて」の面倒くささ

Slackに画像を貼る → スレッドでコメントが分散する → 「どのバージョンの話？」
ちゃんとやるなら、URLを1つ発行してコメントもそこに集めたい。

でもそれを自前で作るとなると、認証・保存・コメントAPIと、急に「アプリケーション」が1個増える。 design-share はこれをどう発行しているのか——今日話すのはその裏側のインフラ。

CloudFront Function（viewer-request）1本が、リクエストが来るたびにこれを判定する。

許可されるのはGET/HEADと、/comments/{slug}/*.json宛のPUTだけ。 他の書き込み系メソッドは経路を問わず即403——多層防御をコード1本に握らせている。

07 仕掛け②

KeyValueStoreが担う、たった数種類のキー

DynamoDBの代わりに使っているのは、CloudFrontに直結できるキーバリューストア。持つ状態はこれだけ。

  • token:{slug}そのパターンの現在の有効トークン。書き換えるだけで即失効=ローテーション
  • "DISABLED"無効化フラグ。トークン値の代わりにこの文字列が入る
  • project:token共有ギャラリー（全体横断閲覧）の共通トークン
  • g:{gslug}名前付きギャラリー（プロジェクト）ごとのスコープトークン
  • pg:{slug}パターンがどのギャラリーに所属するか（最大3件）

読み取りは通常1回、ギャラリー横断の複雑な経路でも最大6回。 テーブル設計もIAMロールも要らない——キーが数種類、それだけ。

08 仕掛け③

コメント機能、DBなしでどう動く？

「投稿」も「一覧」も、正体はS3への素朴なオブジェクト操作に書き換えているだけ。

  1. 投稿: ブラウザが PUT /comments/{slug}/{id}.json を叩く。edge-gateがキー形状とcontent-typeを検査してからS3へ素通し
  2. 一覧: GET /comments-list/{slug} を、edge-gateが内部で ListObjectsV2 ?prefix=comments/{slug}/ に書き換える
  3. 保存: 実体はS3オブジェクトが並んでいるだけ。読み出し方をCloudFront Functionが偽装している

「DBが要る」と思っていた処理の実態は、prefix検索とオブジェクトPUTだった、というだけのことも多い。

09 Skillとの組み合わせ
