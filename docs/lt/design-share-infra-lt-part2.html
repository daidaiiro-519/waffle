01 今日見る9個

「CloudFront + S3」は、9部品の略称だった

前回のスライドでは2サービスと言ったが、実際にcloudformation.yamlを開くと 9種のAWSリソース定義が並んでいる。今日はこれを、リクエストが流れる順に見ていく。

1S3 Bucket 2Origin Access Control 3CloudFront Distribution 4Cache Policy 5Origin Request Policy 6Response Headers Policy 7CloudFront Function 8KeyValueStore 9CloudFormation Stack

どれも単体では地味な部品だが、組み合わせると認証つき・コメント可・キャッシュ制御ありのWebアプリになる。

02 部品① S3

完全に非公開の倉庫

design-shareが扱うデータは全部この1バケットに入っている。ただし外からは1バイトも直接読めない。

PublicAccessBlockConfiguration
BlockPublicAclstrue
BlockPublicPolicytrue
IgnorePublicAclstrue
RestrictPublicBucketstrue
静的Webサイトホスティング未設定
CORS設定なし

4項目すべてtrue＝パブリックアクセスの穴を一切開けない設定。 CORSも定義していない。理由は単純で、コメントの読み書きは配信ドメイン自身からの同一オリジンfetchだけで完結し、 クロスオリジンでアクセスされる想定がそもそも無いから。

  • p/{slug}/index.htmlパターンHTML本体
  • comments/{slug}/{id}.jsonコメント。1件＝1オブジェクト
  • design/{slug}.mdデプロイ元のDesign.md
  • gallery/ ・ galleries/{gslug}.json共有ギャラリー・名前付きギャラリーの実体
03 部品② Origin Access Control

CloudFrontだけが持つ、S3への鍵

S3は完全非公開なのに、CloudFront経由なら読める。その橋渡しをしているのがOAC（Origin Access Identityの後継）。

OriginAccessControlConfig
OriginAccessControlOriginTypes3
SigningBehavioralways
SigningProtocolsigv4

CloudFrontがS3へリクエストする際、毎回SigV4署名を付ける。S3側はBucket Policyで 「署名が正しく、かつ自分自身のDistribution ARNから来たリクエストだけ」を許可する——他人のCloudFrontが同じOACを騙って このバケットを覗くconfused deputy攻撃を、Condition句のARN一致で塞いでいる。

Sid許可アクション範囲
CloudFrontReads3:GetObjectバケット全体
CloudFrontListCommentss3:ListBucketcomments/*のprefix一覧のみ
CloudFrontPutCommentOnlys3:PutObjectcomments/*配下のみ。DeleteObjectは無し

全リクエストの入口、経路ごとに別ルール

1つのDistributionの中に、パスパターンで振る舞いが変わる3つの経路がある。

05 部品④⑤⑥ 3枚のポリシー

挙動を決めているのは、3枚の小さな設定

Cache Policy（自作・no-cache）
DefaultTTL0
MaxTTL1
MinTTL0

CDNの本業である「キャッシュして高速化する」を、design-shareは意図的に捨てている。 トークンの失効やコメントの新着を即座に反映したいので、毎回オリジン（S3）まで見に行かせる。 ここでのCloudFrontの役割は「キャッシュ」ではなくグローバルなリクエストルーター兼TLS終端。

Origin Request Policy（コメントPUT専用・自作）
HeaderBehaviorwhitelist
Headerscontent-type のみ

S3オリジンはAWSマネージドの汎用ポリシー（AllViewer等）を受け付けないため、専用に1枚作った。 x-amz-content-sha256は含めない——OACの署名処理がネイティブに扱うので、ここに書くとむしろ 「許可されないヘッダ」としてCloudFrontに拒否される、という実装上の落とし穴込みの1枚。

3枚目のResponse Headers PolicyはAWSマネージドのSecurityHeadersPolicyをそのままアタッチするだけ。 nosniff・frame-deny等のセキュリティヘッダーを、自作コード無しで全レスポンスに付与できる。

06 部品⑦ CloudFront Functions

edge-gate.js — 10KBの中に、認可ロジック全部

前回の主役。今回は数字で改めてサイズ感を見る。

291行ソースの行数（コメント込み）
8.6KB行コメント・空行を除いた実配置サイズ
10KBcf2ランタイムの上限。約1.4KBの余白

トークン照合・無効化判定・PUTの形/content-type検査・コメント一覧へのリクエスト書き換え・ギャラリーのスコープ判定—— 通常はLambda@EdgeやLambdaで書きそうなロジックが、10KB制限のjsランタイム1本に収まっている。 デプロイ時はds.py update-functionがコメントと空行を削ってから反映する。

07 部品⑧ KeyValueStore

書き込みは、ETagで守られている

DynamoDBを消したはずなのに、DynamoDB的な発想は結局ここに残っている。

token:{slug}のローテーションも無効化も、実体はこのETag一致確認つきの書き込み。 「DynamoDBを使わない」と決めた設計が、結局はDynamoDBが得意とする考え方（条件付き書き込み）を 別の場所でそのまま再発明している——地味だが面白い発見。

08 部品⑨ CloudFormation

1枚のYAMLが、8つの部品を全部知っている

ここまでの8部品（S3〜KeyValueStore）は、すべて1つのCloudFormationテンプレートの中で定義されている。

  • Outputs.DistributionDomain共有URLのベースドメイン
  • Outputs.BucketNameOutコンテンツ保管バケット名
  • Outputs.TokenStoreArnKeyValueStoreのARN
  • Outputs.EdgeGateFunctionNameedge-gate.jsの反映先

ds.py initが、このOutputsをdesign-share.envというテキストファイルに落とし込み、 以降すべてのコマンドがそれを読む。「テンプレート→Outputs→ENV→CLI」という一本道で、 手作業でARNをコピペする工程が存在しない。
