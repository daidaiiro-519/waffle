# ブレスト: Handoffドキュメントの目的とデザイン

**目的:** spec確定後・実装着手前に作成する`HandoffSchema/v1`ドキュメントが、
そもそも「誰の・何のために」存在するのかを言語化し、そのHTML固定テンプレート
（[[brainstorm-advisor-di-and-subagent-roles]]論点6）のデザインを、目的に
沿った形で決める。
**モード:** アイデア発散
**作成日:** 2026-07-18
**経緯:** design-share Skill（[[brainstorm-self-hosted-artifact-design-skill]]）を
論点6の代替として使えないか検討していた際、既存で手作成した2件のHandoff
Artifact HTML（query-redesign用・verification-gate用）を見返したところ、
両方とも「このspecができるまでの過程（ブレスト→advisorレビュー→spec作成→
Handoff記録→実装）」を表現するUI（process trailストリップ）を自然に入れていた
ことに気づいた。これはHandoffの本質的な役割を反映した設計だったのではないか、
という気づきから、論点6のHTML機構詳細ブレストとは別にHandoff固有の目的論を
先に固める必要があると判断し、ブレストを分離した。

---

## 論点1: Handoffドキュメントは何のために存在するか

### たたき台（AIの初期理解、要検証）

Handoffドキュメントの核心は、「specの結論（決まったこと）だけを渡されても、
なぜその結論に至ったかが分からないと、後続（実装者、あるいはコンテキストの
無い別セッションのAI）が実装時の細かい判断に迷う」という問題への対処である。

- `designViewpoints`/`implementationViewpoints`/`constraints`は、ブレスト〜
  advisor批評〜spec確定までの検討過程を凝縮した「圧縮された思考の軌跡」である。
  spec自体（結論）の再掲ではない。
- 手作成した2件のArtifactに共通して現れた「process trail」（01 ブレスト→
  02 advisorレビュー→03 spec作成・修正→04 Handoff記録[現在地]→05 実装、という
  ストリップ表示）は、この「圧縮された過程を渡している」という性質を視覚的に
  裏付けるための装飾だった。
- これは論点10（Handoffが実装に進めるほど「十分」と言える条件）とも関係が
  ある。「本当にadvisorがレビューしたのか」「どのフェーズの成果物として今
  読んでいるのか」が一目で分かることは、十分性判断の材料の一つになりうる。

### 収束（合意事項・2026-07-18）

たたき台のまま確定。その後の論点4（トレーサビリティ・常に最新の要件）・
論点5（デザイン確定）の議論で、この理解（specの結論だけでは後続がWhyを
追えないためHandoffが要る）が前提として一貫して使われ、特に反論が無かった
ため、明示確認のみ行い収束とする。

## 論点2: 「後続」とは具体的に誰・どんな状況を想定するか

process trailやdesignViewpointsの粒度は、実際に読む相手が誰かで変わる。
候補:

- (a) 同じセッション内で、直後にimplementation役割へ引き継ぐAI自身
- (b) 別セッション・別日に実装を再開するAI（コンテキストが完全にリセットされる）
- (c) 人間のレビュアー（実装に進めてよいか最終承認する）
- (d) 将来この機能を保守する別の開発者（AIとは限らない）

想定読者によって「省略してよい情報」「絶対に残すべき情報」が変わるはずだが、
現状のHandoffSchema/v1は読者を単一に固定していない。

### 収束（合意事項・2026-07-18、ddd-advisor/tech-lead-advisor並列相談）

両advisorに対立点はなく、補完的に一致した。

- **読者は単一に絞らず、成果物の種類で分ける**: document.json＝AI向け
  （主に候補2＝コンテキストリセット後の別セッションAI）、HTML＝人間向け
  （主に候補3＝承認可否判断者）
- **候補4（将来の保守開発者）は独立設計不要**: document.jsonの永続化・
  diff更新設計（論点4）により「将来時点の候補2」に収束する
- **Handoffと「ブレスト完了サマリー」（論点6）は主読者が異なる**: Handoff＝
  実装者AI（候補1+2統合）、サマリー＝人間レビュアー（候補3）。これは別々の
  境界づけられたコンテキストであり、無理に統一しない
- **HTMLはdocument.jsonからの一方向派生物**（人間向けDTO）とし、編集可能・
  書き戻し可能にはしない（document.jsonが唯一のsource of truth）
- **AI読者特有の要求水準**（ddd-advisorの追加考察）: コンテキストリセット後
  のAIは会話の暗黙知を一切持ち込めないため、Handoffのユビキタス言語は
  「人間同士が通じればよい」水準では不十分で、暗黙の前提を排した完全な
  明示性が要求される

これで本ブレストの論点1・2・4・5・6が収束。残るは特に無し（本ブレストの
目的であった「Handoffの目的言語化」「HTML固定テンプレートのデザイン確定」
は完了）。

## 論点4: Handoffドキュメントの管理方式（document.jsonの永続化 vs 使い捨て）

### 収束（合意事項）

**document.jsonは永続化する**（specが修正されるたびに使い捨てで再生成する
方式ではなく、既存の`documentId = handoff-{specRefのdocumentId}`命名規約を
使って同一documentへdiff更新し続ける）。理由:

1. **トークン効率**: `scaffold fill`による差分更新は、都度ゼロから組み立て
   直すより明らかに安い。
2. **既存命名規約がトレーサビリティを既に解決している**: `documentId =
   handoff-{specRefのdocumentId}`という規約により、フォルダ構成・specとの
   紐付けは自動的に保証される。document.json永続化＝紐付け管理コスト増、
   という直感は誤りで、実際には命名規約だけで解消済み。
   **2026-07-18訂正**: この規約は「specRef＝今回の作業単位」という前提を
   暗黙に置いていた。usecase specは元々1spec=1機能の粒度なのでこの前提が
   成立するが、`agg-schema`のような集約spec（境界・不変条件のガバナンス
   範囲であり、配下の複数の無関係な変更に共通で参照されうる）ではこの前提が
   崩れる。正しい一般化は次の通り: **documentId＝実際に変更される対象**
   （usecase specの場合はspecRefのdocumentIdと一致する特殊ケース。
   aggregate specの場合は配下の実ファイル、例:
   `handoff-agentschema-v2`）、**specRef＝その変更を拘束するspec**
   （ガバナンスの参照先、今回なら`agg-schema`）。documentIdとspecRefは
   常に一致する必要はない。既存3件のHandoff（usecase spec対象）はこの
   一般化でも変わらない。将来実装予定のHook（spec側のmtime/ハッシュ比較で
   ズレを検知する構想）も、比較対象をspecRefのspecではなくdocumentIdが
   指す実対象にする必要がある点に注意。
3. **将来のモブレビュー/コメント運用**（design-share的な複数人コメント
   フローをHandoffにも持ち込みたい場合）を見据えると、消えない実体
   （document.json）がある方が土台として扱いやすい。

「常に最新である」という要件（本ブレスト冒頭の気づき）は、document.jsonを
一時データ化することでは**なく**、**「spec修正時にHandoff-authoringを
再実行し、同じdocumentIdへdiff更新することを運用ルールとして強制する」**
ことで満たす。これは保留中の**process-reliability論点3**（手順の再現性・
冪等性のHooks拡張）と同種の話であり、将来的にHookで機械的に強制できる
可能性がある（例: spec側のmtime/ハッシュとHandoff側の記録を比較し、ズレて
いればブロック/警告する）。

**役割分担のまとめ**:
- document.json（Handoff） = 永続・diff更新・documentId規約でspecと1:1紐付け
- 「常に最新」の担保 = 運用ルール＋将来的なHook（process-reliability論点3の拡張）
- HTML = document.jsonからのレンダリング成果物（論点6）。人間が実際に見て
  判断する成果物であり、判断根拠のスナップショットとして残す価値がある。

## 論点5: HTML固定テンプレートに何を「必ず」含めるべきか

論点6で確定した大枠レイアウト（00 完成イメージ→01 レビュー状況→
02 設計観点/実装観点/制約）に対し、process trail（過程の可視化）を正式な
構成要素として追加すべきか、それとも02のセクション内で十分表現できて
いるので別枠は不要か。

### 収束（合意事項・2026-07-18、視覚プロトタイプによる決定）

Pattern A〜Gの試作往復（design-shareの「言葉だけで決めず試作artifactの
視覚確認を経る」原則を踏襲）を経て、以下で確定した。

- **process trail（フェーズ進行状況）は正式な構成要素とする**が、独立した
  セクションにはせず、**01 レビュー状況に統合**する（advisorカバレッジと
  並べて表示）。
- **00 完成イメージ**は対象成果物（usecase等）自体の構造・振る舞い図とし、
  図中の各要素に番号バッジを振り、下に置く「読み方」（番号付きステップの
  文章説明）と1対1対応させる。図と文章説明は別カードとして構造的に分離する。
- **配色**: ニュートラル（暖色寄りのpaper背景＋グラファイトink）＋
  アクセント1色のみ（フォレストグリーン系、`#2f5c46`／dark時`#82c4a4`）。
  advisorの色分けタグは廃止し、文字ラベルのみで区別する（色を使いすぎない）。
  アクセントは「現在地」「タブの選択状態」「番号バッジ」等、意味を持つ箇所
  にのみ使う。
- **細部の作り込み**: 角丸（カード16px・図中の箱10px）・微細なシャドウ・
  ピル型タブ等、認知負荷を下げるための質感は色数を絞っても維持する
  （「認知負荷を下げる＝装飾を削る」ではないという指摘を反映）。
- **可読性**: 本文（`consideration`等の長文）は行長62ch程度に制限し、
  行間を広め（1.75）に取る。
- **堅牢性**: 長いテキストによる見切れを防ぐため、SVG内の図ラベルは
  `<text>`ではなく`foreignObject`＋HTMLで折り返し可能にする。メタ情報・
  タブラベル・レビュー行にも`overflow-wrap`を適用。表を今後追加する場合に
  備え、`overflow-x: auto`のスクロール用ユーティリティを用意する。

最終確定版のプロトタイプ: `handoff-uc-check-verification-gate`を題材にした
Pattern G（フォレスト・番号連動版）。ただしこれはArtifactツールで作った
**試作**であり、論点6の合意（Artifactツール不使用）により、実際の固定
テンプレートはWaffleネイティブのHTML render機構として別途実装する
（この試作はその際のデザイン仕様書として使う）。

**02タブのラベル統一（2026-07-19、論点6のブレスト→specハンドオフと合流して確定）**:
「ブレスト→specハンドオフ」の試作を作る過程で、02タブの構成が
「前工程で何が決まったか」「次工程はどう進めるべきか（方針）」「次工程が
必ず守るべきこと（申し送り）」という3層構造に一般化できると判明した。
spec→実装Handoffの既存タブ名（設計観点／実装観点／制約）も、実質この
3層と対応していたため、以下のラベルへ統一する。

| 旧ラベル（spec→実装Handoff） | 旧ラベル（ブレスト→specハンドオフ） | 統一後 |
|---|---|---|
| 設計観点 | 論点の結論 | **前工程の結論** |
| 実装観点 | （無かった。今回新設） | **次工程の方針** |
| 制約 | specへの申し送り | **次工程への申し送り** |

「次工程の方針」は、前工程の結論（なぜそうなったか）と区別される
**次工程がどう進めるべきかの指針**（例:「usecase側はready/blocked/
needs_humanの離散ステータス契約にし、判断ロジックをusecase側に持たせる」）
を指す。この区別（結論の理由 vs 次への指針 vs 必達の制約）は両Handoff種で
共通の骨格になる見込み。

## 論点6: ブレスト→spec作成の間の人間向けチェックポイント文書の要否

「specがどんな形になりそうかを、書き終わる前に人間が把握できるべきでは
ないか」というユーザー指摘から発生した論点（[[brainstorm-advisor-di-and-
subagent-roles]]論点6のHandoff HTML機構の議論とは別件）。

### たたき台

- 現状、ブレスト中の対話（「その前にちゃんとお互い合意してから」の運用
  ルール）がこの役割を代替しているが、チャット上の一過性のやり取りであり、
  Handoffのように後から見返せる・レンダリングされた成果物としては残って
  いない。
- HandoffSchemaの`designViewpoints`/`implementationViewpoints`/
  `constraints`という構造はspec→実装の橋渡しに特化した形であり、そのまま
  転用はできない。ブレスト→spec間に必要なのは「ここまでの合意・spec化
  される見込みの要約」を示す、性質の異なる軽量な文書と考えられる。
- 本ブレスト（Handoffのデザイン確定）が完了してから改めて着手する。

### 収束（合意事項・2026-07-18）

**価値あり、と確定。** ユーザーコメント「結局どういうブレスト結果になった
のかがいつもわからない」が決め手。これはHandoffが解決しようとしている
問題（[[brainstorm-advisor-di-and-subagent-roles]]論点6の発見2:
「人間がspecを見て確認するという行為自体が失われた」）と**全く同じ構造の
問題**が、ブレスト文書側でも起きているということ。既存の「確定事項」
セクションで情報自体は残っているが、たたき台・論点ごとの発散・✏️欄・経緯
等に埋もれ、認知負荷が高く「結局何が決まったか」が一目で掴めない。問題は
情報の不在ではなく、人間向けの合成・視覚化の不在。

**結論**:
1. **価値**: あり
2. **粒度**: ブレストの「確定事項」を生のまま出すのではなく、Handoffと
   同じ問題意識で人間向けに合成・視覚化したサマリーにする
3. **タイミング**: ブレスト完了時に1回

**位置づけ**: これは「Handoff HTML機構」とは別の文書種だが、**同じ機構の
兄弟**（対象がspec→実装ではなく、ブレスト→specという違いだけ）。将来的に
render機構を実装する際、Handoff用に確立するkind別テンプレートの仕組み
（[[brainstorm-advisor-di-and-subagent-roles]]論点6）を、ブレスト完了
サマリーというもう1つのkindとして横展開できる可能性が高い。スキーマ自体は
別途要検討（HandoffSchemaの流用ではなく、ブレスト固有の構造を持つ新規
schemaになる見込み）。

## 訂正: 「00 完成イメージ」の意味の取り違え（2026-07-18）

本ブレストの当初の試作（Pattern A〜D）で、「00 完成イメージ」に
**プロセスの現在地**（ブレスト→advisorレビュー→spec作成→Handoff→実装の
どこにいるか）を描いていたが、これは誤り。

[[brainstorm-advisor-di-and-subagent-roles]]論点6（2026-07-15確定）では、
**「00 完成イメージ」は対象成果物そのものの構造・振る舞いを示す図**と
既に確定していた（Backend/Domain系→集約構造図、Frontend/UI系→UI
ワイヤーフレーム、Platform系→インフラ図）。今回作った試作群は、この
既存合意から逸脱していた（Artifactツール不使用の合意を逸脱していたのと
同種のミス）。

**訂正後の構成**:
- **00 完成イメージ**（確定済み定義に復帰）: 対象usecase自体の構造・
  振る舞い図。`uc-check-verification-gate`の例なら、入力
  （testResultsPath＋spec）→内部でCheckScenarioDriftを呼び出し→優先順位
  判定ロジック→status/reasons出力、という実際のロジック構造を図示する。
- **01 レビュー状況**（確定済み・不変）: advisorカバレッジ。ただし
  「フロー中の現在地」表現に価値を感じたというユーザーコメントを踏まえ、
  このセクション内でフェーズ進行状況も分かりやすく統合表示する
  （新規セクションとしては追加しない）。
