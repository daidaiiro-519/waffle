# ブレインストーミング: ソースコードからUIをSVGで完全再現し、操作マニュアルをコードベースで自動生成し、エンドユーザーが自然言語で修正できる仕組み

**目的:** スクリーンショットを手作業で撮る手間を解消するため、「UIのSVG完全再現」「マニュアルのコードベース生成」「自然言語での修正」が技術的に成立するか、成立するならどの方式が現実的かを探索する。
**モード:** アイデア発散

---

## アイデアダンプ

1. 実レンダリング + DOM→SVGシリアライズ方式（`dom-to-svg`等）でUIを完全再現する
2. Storybookの各storyを自動キャプチャしてSVG化する
3. Playwright操作シナリオ = テスト = マニュアルソースとする「Docs as Tests」パターン
4. UIコンポーネントのProps差分だけを自然言語で吸収するテンプレート機構
5. デザイントークンを唯一の源泉にしてSVGとコードを両方生成する（source of truth統一）
6. スクリーンショットではなくCSS+DOM構造を保持したインタラクティブSVG（ホバー注釈等も表現）
7. コード変更検知でマニュアル画像を自動再生成するCI連携（ドリフト防止）
8. LLMがコード差分を読み、マニュアルの該当ステップ文章だけを自動更新する
9. 自然言語コマンドをUIコンポーネントのpropsパッチに変換するchat-to-JSONパイプライン
10. Waffleのdesign-share Skill資産（自己完結アセット層・埋込フォント）をマニュアル用SVG基盤にも流用する
11. マニュアルの各ステップをMDX化し、SVGを埋め込みつつ注釈テキストをコードで管理する
12. Visual regressionツール（Percy/Chromatic）のスナップショットをそのままマニュアル素材に転用する
13. ブラウザ拡張で実際の操作を記録し、自動でステップ分解＋SVGキャプチャする（操作ログ駆動）
14. 静的解析（AST）からコンポーネントツリーを抽出し、デザイントークンでSVGをレンダリングする（ヘッドレスブラウザなし）
15. エンドユーザー向けには自然言語→編集提案（diff）を人間が承認するHITL（Human-in-the-loop）フロー

**絞り込み候補:**
- **1. 実レンダリング+DOMシリアライズ方式** — 「完全再現」要求を満たせる唯一の現実的な技術的中核。実現性高・インパクト高。
- **3. Docs as Testsパターン** — マニュアルとテストのソース統一により、手作業のスクショ撮影を根本から無くせる。実現性高・インパクト高。
- **15. HITL diff承認フロー** — 自然言語修正を「即時反映」ではなく「提案→人間承認」にすることで安全性と実現性を両立できる。実現性中・インパクト高。
- **7. CI連携での自動再生成** — 作って終わりでなく、コード変更のたびにマニュアルが陳腐化しない運用を担保する。実現性高・インパクト中。
- ~~10. Waffle design-share資産の再利用~~ — 論点1の合意によりWaffle非依存の方針となったため撤回。

## 論点 1: 「完全再現」のためのSVG化は、静的解析ベースか実レンダリング+シリアライズか、どちらを採用すべきか？

### AI 初期見解
**見解:** 実レンダリング（headless browser / Storybook等で実際にコンポーネントを描画）した上でDOMをSVGにシリアライズする方式（`dom-to-svg`等）を採用すべき。静的解析（ASTからコンポーネントツリーを抽出してSVGを直接組み立てる方式）は採用しない。

**根拠:**
- 「全く同じUIデザイン」という要求水準に対し、静的解析はflexbox/gridの実レイアウト計算・実フォントメトリクス・動的データによる表示分岐を再現できず、本質的に「近似」にしかならないから
- `dom-to-svg`等は実際にレンダリングされたDOMのCSS計算値をそのままSVGに転写するため、フォント・色・レイアウトが実物と一致することが構造的に保証されるから
- Storybook/Playwright等、UIコンポーネントを単体表示できる基盤が既にあるプロジェクトなら、追加インフラコストが小さく既存のテスト基盤に相乗りできるから（Waffleのdesign-share Skillも実レンダリング前提の資産を持つ）

### ユーザー見解
> ✏️ これを使用する環境はWaffleを前提としたくない。Waffleに準拠したものでなくてもいい。

### AI 再考見解
**見解:** 方式Bの結論（実レンダリング+DOMシリアライズを採用する）自体は維持できる。根拠3点目の「design-share Skillへの相乗り」は撤回し、Waffleに依存しない一般的な理由に差し替える。方式Bで使う`dom-to-svg`・Playwright/Puppeteer・Storybookはいずれも汎用のOSSで、Waffle固有の資産ではないため、Waffle非依存という制約を置いても結論は変わらない。
**根拠:**
- headless browserによる実レンダリングとDOMシリアライズは、React/Vue/素のHTML等どのフロントエンド技術スタックでも通用する汎用的な手法であり、特定プロダクトの資産に縛られないから
- むしろWaffle資産への依存を外すことで、このスキル／ツールを独立した汎用マニュアル生成の仕組みとして設計する方針が明確になる（論点4で改めて扱う）

### 合意決定
**決定:** 「完全再現」のSVG化は実レンダリング+DOMシリアライズ方式（`dom-to-svg`等の汎用OSS、Waffle非依存）を採用する。
**理由:** 静的解析では動的状態・実レイアウト・実フォントメトリクスを再現できず「完全再現」要求を満たせない一方、実レンダリング方式は技術スタックを問わず適用できる汎用的な解であるため。
**次のアクション:** 論点2（マニュアル生成のアーキテクチャ）に進む。

## 論点 2: 操作シナリオとSVGキャプチャをどう対応づけ、どこをコードのソースとするか？

### AI 初期見解
**見解:** 「操作シナリオスクリプト（Playwright等）」を唯一のソースとし、そこから (a) 各ステップでのSVGキャプチャ、(b) ステップの説明文（見出し・注釈）の両方を生成する構成にすべき。マニュアルのMarkdown/MDXファイルを人間が直接編集する構成にはしない。

**根拠:**
- スクリーンショットとテキストを別々に手作業で同期させると、コード変更時に片方だけ更新されて再びドリフトする（当初の課題「毎回キャプチャを撮る手間」が別の手間に置き換わるだけ）
- Playwrightの操作コード自体が「クリックする」「入力する」といった意味のあるアクション単位を持っているため、そのアクションをステップの説明文テンプレートに機械的にマッピングできる（例: `page.click('#submit')` → 「送信ボタンをクリックします」）
- シナリオスクリプトはテストとしても実行可能なため、「マニュアルの画面が実際の挙動と一致していること」をCIで継続的に検証できる（論点候補7のCI連携と直結する）

### ユーザー見解
> ✏️ プレイライトするにはソースコードにもIDないとできないんじゃない？

### AI 再考見解
**見解:** シナリオスクリプトを唯一のソースとする方針自体は維持するが、「対象アプリのソースコードに一切手を入れない」という暗黙の前提は撤回する。安定したセレクタ（`data-testid`等）が無いアプリでは、最低限のセレクタ付与という形でソース側への軽い介入が導入コストとして発生することを明記する。
**根拠:**
- Playwrightのセレクタにはrole/textベース（ソース変更不要）、CSS/XPath（変更不要だが壊れやすい）、`data-testid`（安定するがソース変更が要る）の3段階があり、行儀の良いアプリでなければ最後の手段に頼らざるを得ないから
- この導入コストを隠さず明示することで、「マニュアル作成のためだけにアプリのソースを触る」ことへの心理的・運用的ハードルを事前に見積もれるようにする必要があるから

### 合意決定
**決定:** シナリオスクリプトをソースとする方針は維持。マニュアルは対象アプリの機能全体をカバーする前提のため、`data-testid`の付与もスコープ限定ではなく**AI支援によるコードベース全体への一括付与**で行う。あわせて、新規コンポーネント追加時に`data-testid`漏れが発生しないよう**lintルールでの継続担保**を前提条件に含める。
**理由:** 一括付与自体はAST走査による機械的パターンマッチで検出できる定型作業であり、ディレクトリ単位で並列化すればAIにとって低コスト。真のコストは命名規約の設計・初回レビュー・継続運用（lintルール）に移るため、そこを前提条件として明記した上で「ありの前提」とする方が、後で導入コストの過小評価による手戻りを防げる。
**次のアクション:** 論点3（自然言語修正の対象範囲・反映方式）に進む。

## 論点 3: 自然言語修正の対象範囲は「マニュアル文章のみ」か「UI/デザイン自体」まで含むか？また即時反映かHITL承認か？

### AI 初期見解
**見解:** 自然言語修正の対象は「マニュアルの文章・構成（ステップの説明、注釈、章立て）」に限定すべきで、「UI/デザイン自体（ソースコードの見た目）」の修正は対象外とする。反映方式は即時反映ではなくHITL（提案→人間承認→反映）とする。

**根拠:**
- UI自体を自然言語で直接書き換えることは、実質的にプロダクトのソースコードを自然言語で改変することと同義であり、マニュアル生成という当初のスコープを大きく超えてプロダクト開発そのものに踏み込むことになるから
- 論点1・2の合意（実レンダリング+シリアライズ、シナリオスクリプトがソース）を踏まえると、UIの見た目はソースコードとレンダリング結果に対して一意に決まる派生物であり、そこを自然言語で直接編集可能にすると「どちらが正か」の二重管理が発生するから
- マニュアルの文章側であれば、エンドユーザー（非エンジニアの文書担当者等）が「この説明をもっと簡潔にして」「この画面の前に注意書きを足して」のような修正を、既存のシナリオスクリプト＋生成テキストのテンプレートに対する差分として安全に扱え、かつ即時反映ではなく承認フローを挟むことで生成物の暴走（意図しない改変の混入）を防げるから
